## 1. Executive summary

**The question is not "which pattern is better" but "where does control live, and how much must agents see each other's work."** An *orchestrator* (supervisor / hub-and-spoke) centralizes control in one agent that decomposes a task, routes typed subtasks to workers in fresh context windows, and synthesizes their returns; coordination happens only at decomposition and aggregation. A *blackboard* removes explicit dispatch: agents read from and write to one shared, structured state, and a control unit re-selects agents whenever the board changes, so a single write can re-trigger dependents mid-flight. Everything downstream — tokens, latency, auditability, failure surface — flows from that one structural difference: **message-passing + explicit routing vs. shared state + reactive re-triggering.**

**An honest caveat frames the whole comparison: on today's frameworks both are largely turn-based.** The differentiator is not literal simultaneity but the coordination *substrate*. (Note on sourcing: a "one-LLM-at-a-time / prompt-routed" phrasing is sometimes attributed to arXiv 2606.10662 (DeLM), but that paper actually argues the opposite — for decentralized, asynchronous, parallel agents — so the turn-based caveat should be stated as reasoning about current mainstream frameworks, not pinned to DeLM.)

**The evidence does not crown a universal winner.** Anthropic's orchestrator-worker Research system beat single-agent Claude Opus 4 by 90.2% on breadth-first research and cut research time up to 90% via 3–5 parallel Sonnet 4 subagents under an Opus 4 lead, but explicitly fails on interdependent work like most coding, where agents must share context. Blackboard systems (LbMAS, 81.68% avg across six benchmarks at ~3.5× fewer tokens; the data-science volunteer-agent system at 13–57% end-to-end gains) win on interdependent, emergent, consensus-seeking work. LangChain's τ-bench-with-distractors benchmark even found a decentralized swarm slightly beat a supervisor while using fewer tokens, because the hub must restate subagent output.

**Three forces reshape this comparison for a 2026 reader and are woven through the sections below.** (1) *Reasoning models* partly substitute for fan-out: because multi-agent's advantage is largely that it spends more tokens (token usage alone explained ~80% of performance variance in Anthropic's eval), a single reasoning agent with an adaptive thinking budget can absorb work that once justified parallel subagents — with compute-normalized studies now showing single agents matching or beating multi-agent systems on multi-hop reasoning under equal token budgets. (2) *Protocol substrate* determines how either topology is physically wired: MCP is intrinsically hub-and-spoke (favoring orchestrators), A2A is peer messaging that can wire either shape but provides no native blackboard — a shared board requires adding a broker/shared-state layer. (3) A *bounded/controlled blackboard* — a reactive shared state governed by a deterministic control unit and gate — is an under-explored but well-motivated design that borrows the orchestrator's discipline without its message-restatement tax.

The practical default that the field converged on by 2026: use a single reasoning agent for dependent work; use an orchestrator that *owns context* for cleanly decomposable/breadth-first work; reserve genuine shared-state collaboration for interdependent, cross-checking tasks — under strict iteration caps and, ideally, a deterministic gate.

## 3. The orchestrator (supervisor) pattern

### Definition & Mechanics

In the orchestrator (supervisor / hub-and-spoke) pattern, a central LLM agent owns control flow. It "dynamically breaks down tasks, delegates them to worker LLMs, and synthesizes their results" [1]. The defining property—distinguishing it from *routing* (fixed classification into predefined branches) and *parallelization* (pre-defined independent subtasks)—is that subtasks are **not pre-defined but determined by the orchestrator based on the specific input** [1]. Mechanically: the user query hits the supervisor, which analyzes it, develops a strategy, and spawns specialized sub-agents to explore different aspects [2]. Each sub-agent runs in its own **isolated, fresh context window**, executes its own tool loop, and returns distilled findings; the supervisor aggregates results and decides whether more work is needed [2]. Control follows a strict hub-and-spoke topology: sub-agents never talk to each other, only to the hub. In LangGraph's supervisor library this is implemented via **tool-based handoffs**—the supervisor invokes `create_handoff_tool`, and "control always returns to the supervisor after each subagent completes" [3][4]. History propagation is configurable (`output_mode="full_history"` vs `"last_message"`), and supervisors can nest into **hierarchical teams** (a supervisor of supervisors) [4].

### SOTA Implementations (2025–2026)

- **Anthropic** codified the "orchestrator-workers" workflow in *Building Effective Agents* [1] and productionized it in *How we built our multi-agent research system*: a Claude Opus 4 lead agent spins up 3–5 Claude Sonnet 4 sub-agents in parallel, each with a clean context, plus a dedicated CitationAgent for attribution [2].
- **LangGraph** ships a first-class `langgraph-supervisor` library with handoff tools, configurable message forwarding, and multi-level hierarchical teams [3][4]. LangChain now also recommends implementing the supervisor directly via tool-calling for finer context control [3].
- **OpenAI Agents SDK** (March 2025, successor to Swarm) offers two flavors: *agents-as-tools* (`agent.asTool()`, manager stays in charge and synthesizes—true centralized orchestration) versus *handoffs* (`handoff()`, decentralized peer takeover) [5]. The manager/agents-as-tools mode is the supervisor pattern proper.
- **Microsoft AutoGen / Magentic-One** uses a lead **Orchestrator** driving two ledgers: a **Task Ledger** (plan + facts) and a **Progress Ledger** (self-reflection, next-agent assignment). It delegates to WebSurfer, FileSurfer, Coder, and Terminal agents and re-plans when progress stalls [6].
- **CrewAI** hierarchical process: setting `Process.hierarchical` with a `manager_llm`/`manager_agent` makes a manager dynamically allocate tasks; the manager must sit **outside** the worker pool [7].
- **Amazon Bedrock multi-agent collaboration** (GA March 2025) lets you designate a supervisor agent with collaborators, offering "supervisor" and "supervisor with routing" modes (routing bypasses full orchestration for simple requests); GA added Inline Agents and Payload Referencing [8].
- **Google ADK** implements the Coordinator/Dispatcher pattern: a central `LlmAgent` with `sub_agents` uses LLM-driven `AgentTransfer` (configurable transfer scope) or explicit `AgentTool` invocation [9].

### Strengths

Routing clarity and control: a single locus decides who does what, giving strong **auditability**—every delegation is an explicit, traceable step [5][10]. Modularity: specialists have isolated prompts, tools, and policies, so you add capabilities without touching the whole system; OpenAI frames this as improving "capability isolation, policy isolation, prompt clarity, or trace legibility" [5]. Context isolation is the marquee benefit—sub-agents explore in parallel with independent windows, then "condense the most important tokens" back to the lead, effectively compressing far more information than a single window holds [2]. This delivered a **90.2% improvement** over single-agent Claude Opus 4 on Anthropic's internal research eval, cutting research time up to 90% via parallelism [2].

### Weaknesses

The hub is a **single point of failure and bottleneck by design**—if it goes down, everything stops [10]. It adds a reasoning step before every delegation and a synchronous scatter-gather merge, so progress is gated by both the central merge and the slowest worker [11][2]. **Token overhead** is structural: the orchestrator must hold the full task, all worker results, and synthesis context; supervisor reasoning loops can consume 2–3× more tokens than direct patterns, and at 4+ workers the orchestrator can blow past context limits [10]. Most fundamentally, there is **no mid-task cross-agent reaction**: "the lead agent can't steer subagents, subagents can't coordinate, and the entire system can be blocked while waiting for a single subagent" [2]. Coordination happens only at decomposition and aggregation, not during execution [11]. Consequently Anthropic notes domains "that require all agents to share the same context or involve many dependencies between agents are not a good fit"—e.g., most coding tasks [2].

### Token & Latency Behavior (published numbers)

Anthropic's measured figures: agents use ~**4× more tokens** than chat; multi-agent systems ~**15× more tokens** than chat—so the pattern only pays off on high-value tasks [2]. Token usage alone **explained 80% of the variance** in browsing-eval performance, meaning the pattern's win is largely "scaling tokens" [2]. Parallelism (3–5 sub-agents; 3+ concurrent tools each) cut latency up to 90% [2]. Third-party production analyses estimate ~2–3× token overhead versus direct patterns and note throughput ceilings (e.g., a 3s orchestrator call bottlenecking ~20 waiting workers) [10].

### Production Best Practices

Anthropic's guidance: give the orchestrator **detailed delegation instructions** (objective, output format, tools, boundaries)—vague prompts caused duplicate work and over-spawning (once 50 sub-agents for a trivial query) [2]. Embed **scaling rules** so effort matches query complexity [2]. Evaluate with ~20 representative queries early, LLM-as-judge rubrics (factual/citation accuracy, completeness, source quality, tool efficiency, 0.0–1.0), plus human review for edge cases [2]. Because "minor failures can be catastrophic," use **resumable checkpoints**, inform agents of tool errors for adaptive recovery, full tracing, and **rainbow deployments** to update stateful running agents safely [2]. Bedrock adds Payload Referencing to avoid re-embedding data in every hub↔worker message [8]. Emerging "Agent Teams" work adds shared task lists, peer messaging, and file locking to overcome the no-mid-task-coordination limit [11].

**Key findings**

- The orchestrator pattern's defining trait vs. routing/parallelization: subtasks are dynamically determined by the central LLM based on input, not pre-defined [1].
- Sub-agents run in isolated fresh context windows and communicate ONLY through the hub; they cannot coordinate mid-task, which is the pattern's core structural limitation [2][11].
- Anthropic's orchestrator-worker research system beat single-agent Claude Opus 4 by 90.2% and cut latency up to 90% via 3-5 parallel sub-agents [2].
- Token cost is the central tradeoff: agents use ~4x, multi-agent systems ~15x more tokens than chat; token usage explained 80% of performance variance [2].
- The hub is a single point of failure and bottleneck; supervisor loops add ~2-3x token overhead and the orchestrator can exceed context limits at 4+ workers [10].
- All major frameworks converged on this pattern in 2025: LangGraph supervisor, OpenAI Agents SDK (agents-as-tools), AutoGen/Magentic-One ledgers, CrewAI hierarchical, Bedrock multi-agent collaboration (GA Mar 2025), Google ADK coordinator/dispatcher [3-9].
- Anthropic explicitly avoids the pattern for shared-context/high-dependency tasks like most coding [2].

## 4. The blackboard pattern

### Origins and theory

The blackboard architecture originated in the **Hearsay-II** speech-understanding system, developed at Carnegie Mellon between roughly 1971 and 1976 — "the first large blackboard system" [1][2]. Its central metaphor is a group of human experts collaborating around a physical blackboard: each writes partial results others can see, and no expert speaks unless the current state of the board invites a contribution. Formally, a blackboard system has **three parts** [1][6]:

1. **The blackboard** — a global, shared working memory holding input data, partial solutions, alternatives, and final solutions. In Hearsay-II it was organized as a hierarchy of *solution levels* (signal → segment → word → phrase), where hypotheses at each level aggregate or abstract elements at the level below [1].
2. **Knowledge sources (KSs)** — logically independent, self-selecting modules that make "very few assumptions about how their knowledge contributions would be used by other components" [1]. They run as parallel processes "activated asynchronously depending on data events" [1] — i.e., changes to the blackboard.
3. **Control / scheduler** — a data-directed control element that "makes runtime decisions about which knowledge sources to execute next" based on the current board state [4]. The reasoning style is **hypothesize-and-test**, applied *opportunistically and incrementally* rather than in a fixed pipeline [3].

H. Penny Nii's two-part **"Blackboard Systems"** (AI Magazine 7(2) and 7(3), 1986) is the canonical synthesis, defining the blackboard model and tracing its evolution [5]. The pattern spread to **HASP/SIAP** (1973–1975, sonar/ocean-surveillance signal interpretation), which introduced event-based control, and to **BB1** (Barbara Hayes-Roth, Stanford, 1984), which added a *separate control blackboard* so systems could reason about, explain, and learn their own problem-solving strategy [8][9] (from-training: HASP is often dated to SRI/Stanford; sources here place the lineage but disagree on exact institution).

### Modern revival for LLM agents

The pattern has re-emerged in 2024–2026 as an antidote to brittle, hard-wired LLM agent topologies. The core idea: heterogeneous agents read and write **one shared, structured state** — "a shared, structured workspace where heterogeneous agents post partial results, hypotheses, constraints, and goals for others to observe, refine, or refute" [3] — and a control unit re-selects agents based on what's on the board, so a **write can re-trigger dependent agents mid-flight** [7].

Concrete recent systems:

- **"Exploring Advanced LLM Multi-Agent Systems Based on Blackboard Architecture"** (arXiv 2507.01701, 2025) maps the three classical parts directly: blackboard as global DB, LLM agents (planner, decider, critic, conflict-resolver) as KSs, and a control unit that "iteratively selects agents based on blackboard state" until consensus. It reports competitive accuracy (81.68% avg) at **~3.5× fewer tokens** than AFlow (4.72M vs 16.7M on MATH) because shared state removes per-agent memory modules [7].
- **"LLM-Based Multi-Agent Blackboard System for Information Discovery in Data Science"** (arXiv 2510.01285, 2025): a central agent posts requests; subordinate agents "volunteer to respond based on their capabilities," eliminating the need for a coordinator that knows each agent's expertise. It reports **13–57% relative gains** in end-to-end success over RAG and master-slave baselines [10].
- **Terrarium** (arXiv 2510.14312) revives the blackboard specifically for safety/privacy/security studies, emphasizing centralized, auditable communication over direct agent-to-agent messaging [11].

**Relation to BSP/Pregel.** LangGraph, the dominant framework, is explicitly "Pregel-inspired," implementing a **bulk-synchronous-parallel (BSP)** execution model [12][13]. Work is organized into *supersteps*: nodes read the previous superstep's state, execute in parallel, and their writes are merged at a **barrier** before the next superstep [12]. LangGraph's **StateGraph** defines a shared `State` schema whose keys are *channels*, each with a **reducer** deciding how concurrent writes merge — overwrite by default, or accumulate (e.g., `add_messages` appends and de-dupes by ID) [12][13]. This is a modern, disciplined blackboard: the shared state is the board, nodes are KSs, and the barrier-plus-reducer machinery tames write contention that plagued classical designs. Two related coordination ideas recur: **stigmergy** — indirect coordination where agents react to traces left in a shared environment, like ant pheromones [14] — and CRDT-based observation. **CodeCRDT** (arXiv 2510.18893) has agents coordinate purely by observing a shared CRDT document with last-writer-wins convergence, no message passing; a write triggers others to skip completed work, adopt naming, and back off overlapping edits [15].

### Strengths

- **Emergent, opportunistic collaboration** — agents self-select based on board state rather than a fixed script; roles can be added without rewiring [7][10].
- **Mid-flight reaction** — a single write re-triggers dependents, so the system adapts "in a timely manner according to contents of the blackboard" [7][15].
- **Token/context efficiency** — one shared state replaces redundant per-agent memory and pairwise message copies, cutting tokens ~3.5× in one study [7].
- **WORM auditability** — Terrarium appends "all modifications to an immutable log" (write-once/read-many), giving tamper-evident, replayable audit trails [11].

### Weaknesses

- **Control complexity & nondeterminism** — the scheduler is the hard part; "agent behavior may vary unpredictably despite identical inputs" [11].
- **Write contention / races** — concurrent unstructured writes cause collisions where "one agent's update overwrites another's critical data" [16], and CRDT convergence still leaves 5–10% *semantic* conflicts (duplicate declarations, type mismatches) [15].
- **Scalability limits** — observation overhead scales O(N×U); CodeCRDT projects speedup degrading below 1.0× beyond ~25–50 agents, with 82–189% code-volume inflation and a measured 7.7% quality drop [15].

### Best practices

Emerging guidance (esp. **PatchBoard**, arXiv, schema-grounded mutation [16]): (1) **define explicit schemas** for shared state and validate at the mutation boundary, not retrospectively; (2) enforce **atomic, type-safe writes** so updates fully succeed or fail; (3) **scope write access per role** to shrink the collision surface; (4) **log every mutation** with agent identity and timestamp for auditability; (5) use **reducers/barriers** (LangGraph-style) to deterministically merge concurrent writes; and (6) bound the number of superstep rounds to guarantee termination [7][12][16].

**Key findings**

- The blackboard pattern originated in Hearsay-II (CMU, 1971-1976), the first large blackboard system, and has three parts: a shared global blackboard (hierarchical solution levels), independent event-triggered knowledge sources, and a data-directed control/scheduler [1][2].
- H. Penny Nii's two-part 'Blackboard Systems' (AI Magazine, 1986) is the canonical synthesis; BB1 (Hayes-Roth, 1984) added a separate control blackboard for self-reasoning, and HASP/SIAP introduced event-based control [5][8][9].
- 2024-2026 LLM systems revive the pattern: agents read/write ONE shared structured state and a write re-triggers dependent agents mid-flight, replacing rigid master-slave topologies with self-selecting 'volunteer' agents [3][7][10].
- LangGraph implements this as a Pregel/BSP model: supersteps with parallel node execution, barrier synchronization, and per-channel reducer functions (e.g. add_messages) that deterministically merge concurrent writes into shared StateGraph state [12][13].
- Reported strengths: 13-57% end-to-end gains over RAG/master-slave baselines [10], ~3.5x token reduction from shared state [7], and WORM immutable-log auditability [11].
- Reported weaknesses: control complexity and nondeterminism [11], write contention/collisions and residual 5-10% semantic conflicts even with CRDTs [15][16], and scalability degrading past ~25-50 agents with code-volume inflation and quality drops [15].
- Best practice consensus (PatchBoard): schema-grounded, atomic, role-scoped, per-agent-logged mutations validated at the write boundary, combined with reducer/barrier merging and bounded superstep rounds [16][7][12].

## 5. Head-to-head comparison

### The core distinction

An **orchestrator** (a.k.a. supervisor / hub-and-spoke / orchestrator–worker) pattern puts one central agent in charge: it decomposes the task, decides who runs next and what they see, collects results, and produces the final answer. Anthropic's production Research system is the canonical example — a lead agent plans, spawns 3–5 specialized subagents in parallel, and synthesizes their findings with a separate citation pass [1]. A **blackboard** pattern removes explicit dispatch: agents (classically called "knowledge sources") read from and write to a shared workspace, and each one *self-selects* to contribute when the evolving shared state matches its capabilities, without a coordinator holding precise knowledge of every agent's skills [2][3].

The honest caveat up front: **on today's frameworks, both are ultimately turn-based.** Centralized systems are "largely prompt-routed," and even parallel subagents don't share progress in real time — only one LLM can process and communicate at a time when coordination funnels through an orchestrator [4]. So the real differentiator is *not* literal simultaneity. It is **shared state + reactive re-triggering** (blackboard) versus **message-passing + explicit routing** (orchestrator). Everything below flows from that.

### Dimension-by-dimension

**Control flow.** Orchestrator: centralized, explicit, deterministic-ish routing — every hop is a visible decision by the hub [5]. Blackboard: emergent and data-driven — agents trigger on state changes rather than on assignment [2][3].

**State sharing.** This is the crux. Orchestrators pass state *through messages* (or through the hub's context); blackboards keep a *common working memory* all agents read/write [4]. Shared memory enables richer coordination but demands concurrency controls, versioning, and invalidation logic; a recommended mitigation is isolated memory by default with explicit, *versioned publish events* to the shared store at task-completion boundaries [6].

**Context / token cost.** Multi-agent (orchestrator) systems burn ~15× the tokens of a single chat, and token usage alone explains ~80% of performance variance [1]. In the supervisor pattern specifically, the hub must restate subagent outputs, inflating cost; LangChain measured swarm (a decentralized handoff pattern closer to blackboard in spirit) using the *lowest* tokens, with supervisor consistently higher [5]. One implementation reported supervisor consuming ~47% more tokens than swarm (~2,800 vs ~1,900 avg) [7]. Blackboard's self-selection can also reduce redundant routing calls [3].

**Latency.** Every orchestrator delegation adds a round-trip through the hub [8]. Measured supervisor-vs-swarm: single-domain 4.2s vs 2.8s; multi-domain handoffs 9.1s vs 5.4s — roughly a 33–41% reduction by skipping the intermediary [7]. Orchestrator throughput is also capped: if the hub's LLM call takes 3s, 20 waiting workers cap decomposition at ~6.7 tasks/sec [9].

**Determinism / reproducibility.** Neither is deterministic — agents are "non-deterministic between runs, even with identical prompts" [1]. But the orchestrator's single control point makes runs *more* reproducible in practice; blackboards "lack formal guarantees for deterministic convergence under concurrent writes" [6].

**Auditability.** Genuinely contested. The orchestrator view: all traffic through one point yields high observability and clean audit trails, favored for compliance [8][5]. Two opposing readings of blackboard exist: critics say there is "no single trace," so reconstructing a failure means replaying concurrent agents [6]; yet the *Terrarium* work argues the opposite — a shared blackboard is a **single, unified record** of all state changes, easier to instrument than intercepting bilateral messages [10]. The resolution: a *disciplined* blackboard with versioned writes is highly auditable; an *ad-hoc* one is not.

**Fault tolerance / single point of failure.** Orchestrator: the hub is a single point of failure and throughput bottleneck [8][9]; Anthropic mitigates with retry logic and checkpoints, noting "minor system failures can be catastrophic" without them [1]. Blackboard: no central failure point, but emergent failures are "invisible in single-agent testing" and surface only in production [8].

**Scalability.** Orchestrators bottleneck on the hub and degrade as history accumulates — routing accuracy drops after ~8–12 subagent round-trips as old messages crowd out current state [11]. Blackboards scale better when the coordinator can't have full observability of every agent, since agents volunteer [3] — but a fully connected agent set explodes combinatorially (4 agents → 6 failure points, 10 → 45) [11].

**Debuggability.** Orchestrator wins: explicit control flow, centralized traces [5][7]. Blackboard is hard — coordination-type failures dominate; the UC Berkeley MAST taxonomy of 1,600+ traces found inter-agent misalignment and coordination among the top failure clusters, with step repetition (15.7%), termination-unawareness (12.4%), and spec-disobedience (11.8%) common across 7 frameworks [12].

**Mid-task adaptivity.** Blackboard's structural advantage: any agent can react to newly posted state and self-insert, enabling flexible re-triggering [2][3]. Orchestrators adapt only through the hub re-planning, which is more controlled but less responsive.

### Bottom line

Benchmarks don't crown a universal winner. Anthropic's orchestrator beat single-agent Opus by 90.2% on breadth-first research but fails on interdependent work like coding [1]. LangChain found decentralized swarm slightly *beat* supervisor and used fewer tokens on τ-bench-with-distractors [5]. Blackboard papers report 13–57% end-to-end gains for information-discovery tasks [3]. Choose orchestrator for auditable, compliance-bound, tightly-coupled workflows; choose blackboard for open-ended, breadth-first, adaptivity-heavy tasks where you can afford concurrency discipline. For regulated domains (EU AI Act Annex III), the auditable centralized pattern is usually the more defensible choice [6].

**Key findings**

- Both patterns are turn-based on current frameworks; the true differentiator is shared state + reactive re-triggering (blackboard) vs message-passing + explicit routing (orchestrator), not literal parallel execution [4].
- Orchestrator multi-agent systems consume ~15x the tokens of a single chat, and token usage alone explains ~80% of performance variance; the supervisor's need to restate subagent output inflates cost ~47% over decentralized swarm [1][7].
- Anthropic's orchestrator-worker Research system beat single-agent Opus 4 by 90.2% on breadth-first research but explicitly fails on interdependent tasks like coding that require shared context [1].
- LangChain's benchmark on modified tau-bench found decentralized swarm slightly outperformed supervisor AND used fewer tokens, because the supervisor must translate/restate sub-agent responses [5].
- Latency: measured swarm handoffs ran 33-41% faster than supervisor (e.g. 5.4s vs 9.1s multi-domain) by skipping the hub round-trip; orchestrator throughput caps around 6.7 tasks/sec when the hub LLM call is ~3s [7][9].
- Auditability is genuinely contested: one view says blackboards have 'no single trace' [6], while the Terrarium paper argues a shared blackboard is a unified, easier-to-audit record than bilateral message-passing [10]. Disciplined, versioned writes resolve the tension.
- Orchestrator = single point of failure + bottleneck; routing accuracy degrades after ~8-12 subagent round-trips as history crowds out current state [8][11].
- UC Berkeley's MAST taxonomy (1,600+ traces, 7 frameworks) shows coordination/inter-agent-misalignment failures dominate, making blackboard-style systems hard to debug: step repetition 15.7%, termination-unawareness 12.4%, spec-disobedience 11.8% [12].

## 6. When to use each

### When to use each

**Reach for an orchestrator when the shape of the work is knowable up front.** It is the right default for: (1) *routing/triage* — the right handler is identifiable from the input; (2) *independent parallel tool calls* — fan-out/fan-in scatter-gather; (3) *fixed linear pipelines* — the next agent is deterministically defined, not chosen; (4) *breadth-first exploration* — independent parallel search paths whose aggregate context exceeds one window (Anthropic's canonical research win); and (5) anything needing *strict audit, cost caps, and a single traceable control flow*. The orchestrator's marquee real-world advantage is that its single control point is where **human-in-the-loop gating and durable interrupt/resume** (e.g. LangGraph's `interrupt()` / `Command(resume=...)` over a checkpointer) deliver the cleanest approval and recovery story.

**Reach for a blackboard when problem structure emerges during computation and sub-results are interdependent.** It wins for: iterative refinement toward consensus; cross-agent verification (a critic inspecting what peers posted); constraint satisfaction where satisfying one constraint invalidates another; and any case where one agent's output must *immediately* inform others. The volunteer-agent property matters when no coordinator can plausibly know every agent's expertise up front.

**Reach for a bounded/controlled blackboard when you need blackboard-style interdependence *and* orchestrator-style guarantees** — reproducibility, bounded cost, a clean audit trail — which is common in regulated or high-stakes settings (see the hybrid section).

**Reach for neither — a single reasoning agent — when the task is dependent/stateful but not genuinely breadth-first.** Because token spend explains most of multi-agent's edge, an adaptive-thinking-budget single agent captures much of that gain without the coordination tax; escalate to multi-agent only when parallel search paths are real and task value clears the (higher, lumpier) reasoning-token bill.

### Plain-language decision checklist

Ask, in order:
1. **Decomposable into independent subtasks up front?** → orchestrator (parallel/concurrent).
2. **Right handler identifiable from the input alone?** → orchestrator (routing/handoff).
3. **Fixed, ordered pipeline?** → orchestrator (sequential).
4. **Do subtasks depend on each other, or must one agent's output immediately inform another?** → shared state (blackboard) or a single continuous-context agent — *not* isolated workers (the Cognition "Mario-vs-Flappy-Bird" failure).
5. **Need cross-checking, iterative refinement, or consensus?** → blackboard / maker-checker.
6. **Need strict audit, cost caps, reproducibility, HITL approval?** → orchestrator, or a *bounded* blackboard if you also need interdependence.
7. **Open-ended with no known plan?** → single continuous-context reasoning agent, or a small (≤3-agent) collaborative board with a hard round cap.
8. **Could one deep-reasoning agent do this within one context window?** If yes → prefer it; multi-agent only pays off on genuine breadth-first parallelism or aggregate-context-over-window tasks.

**Guardrails that apply to any multi-agent choice:** cap iterations hard (LbMAS caps at 4 rounds, averaging ~2.9–3.3; coding control planes cap auto-fix at 3); keep collaborative group-chat/debate small (≤3 agents) to avoid conformity and non-convergence; and never let an LLM render the final gate — "LLM proposes, deterministic rules enforce, human authorizes rule changes."

### 6.1 Deep dive: criteria & case studies

### The core distinction

The choice between **orchestrator** and **blackboard** patterns is fundamentally a choice about *where control lives* and *how much agents need to see each other's work*. An orchestrator (also called supervisor, lead-agent, or coordinator) is code or an LLM that holds a single point of authority: it decomposes the task, decides who runs next, controls what each worker sees, and synthesizes the final answer. Workers typically run in isolated context windows and report back. A blackboard replaces that central router with a *shared knowledge space*: specialized agents ("knowledge sources") read the current state, post partial results, and a lightweight control unit dynamically selects who acts next based on what is already on the board [1]. Orchestrator = top-down decomposition with accountable central control; blackboard = shared-memory emergence where structure appears during computation [1].

### When the orchestrator wins

The orchestrator pattern is the right default when the *shape of the work is knowable up front*. Anthropic's Research system is the canonical case: a lead agent plans, spawns 3–5 subagents in parallel with self-contained task descriptions and fresh context windows, and merges their findings with a separate citation pass. Each subagent "doesn't know the other subagents exist and cannot coordinate mid-task" [2]. This beat single-agent Claude Opus 4 by 90.2% on their internal eval, and token usage alone explained 80% of the performance variance [3] — meaning the orchestrator wins precisely because it *allocates independent, parallel effort* across a breadth-first search space.

Orchestrators dominate these cases:

- **Routing / triage** — an initial agent classifies a request (billing, technical, account) and dispatches to a specialist. This is Microsoft's *handoff* pattern; use it when "the appropriate agent is identifiable from the initial input" [4][5]. Customer-support routers with 90%+ autonomous resolution are built this way [1].
- **Independent parallel tool calls** — Microsoft's *concurrent* (fan-out/fan-in, scatter-gather) pattern, e.g. four analysts scoring the same stock simultaneously, then aggregating [4].
- **Linear pipelines** — Microsoft's *sequential* pattern, where "the choice of which agent gets invoked next is deterministically defined... and isn't a choice given to agents" [4]. A four-stage contract-generation pipeline (template → clauses → compliance → risk) is the example.
- **Strict control, audit, and cost predictability** — deterministic routing "is the easiest to debug because there's a single control flow to trace"; when the wrong path is chosen you inspect one decision rather than guessing [4]. Central control also caps token spend, important because multi-agent setups burn ~15× the tokens of a normal chat [3].

### When the blackboard wins

The blackboard is the right choice when "well-defined structures or workflows are unavailable" and the problem *structure emerges during computation* [1]. Because every agent reads the full shared state before acting, the pattern natively supports the things orchestrators struggle with: interdependent checks, cross-agent verification, and iterative refinement toward consensus.

Blackboards dominate these cases:

- **Iterative refinement / consensus** — the LbMAS system runs discussion cycles (planner, decider, critic, conflict-resolver) until agents converge, using ~2.88 rounds on easy problems (MMLU) vs 3.29 on hard ones (MATH). It scored 81.68% average across six benchmarks, beating chain-of-thought by 4.33% and static multi-agent systems by 5.02%, while using the fewest tokens among dynamic systems [1].
- **Cross-agent verification** — a dedicated critic or verification agent can inspect what other agents posted, and disagreement can trigger tiebreaking agents for independent verification [6].
- **Constraint satisfaction & emergent problem-solving** — the blackboard "embodies constraint satisfaction through multiple perspectives narrowing the solution space," ideal when the solution path isn't predetermined [6]. Broader surveys report blackboard architectures outperforming RAG and master-slave baselines by 13–57% on end-to-end task success [1][7].
- **Tasks where one agent's output must immediately affect others** — because communication happens through shared state, a new fact posted by one agent is instantly visible to all, overcoming master-slave setups where subordinates "execute instructions even when they hold outdated information" [1].

### The interdependence counterpoint

The strongest argument against naive parallel orchestration comes from Cognition ("Don't Build Multi-Agents"). Their two principles: *(1) share context and full agent traces, not just messages*, and *(2) actions carry implicit decisions, and conflicting decisions carry bad results* [8]. Their famous failure: subagent 1 builds a Super Mario background while subagent 2 builds Flappy Bird pipes — "decision-making ends up too dispersed and context isn't shared thoroughly enough" [8]. This is exactly the case where a blackboard (shared trace) or a single-threaded linear agent beats isolated-worker orchestration. Anthropic agrees on the boundary: most coding tasks "involve fewer truly parallelizable tasks than research," and tasks requiring "all agents to share the same context or... many dependencies between agents" are a poor fit for isolated subagents [2][3]. By 2026 the debate has largely converged: use an orchestrator that *owns context* for decomposable work, keep dependent work in one continuous context, and reserve blackboard/group-chat for genuine collaborative reasoning — capped at ~3 agents to avoid infinite loops [4].

### Decision checklist

Ask, in order:

1. **Is the task decomposable into independent subtasks up front?** Yes → orchestrator (parallel/concurrent). No → keep going.
2. **Can the right handler be identified from the input alone?** Yes → orchestrator (routing/handoff).
3. **Are the steps a fixed, ordered pipeline?** Yes → orchestrator (sequential).
4. **Do subtasks depend on each other, or must one agent's output immediately inform another?** Yes → blackboard (or single-threaded agent).
5. **Do you need cross-checking, iterative refinement, or consensus?** Yes → blackboard / maker-checker.
6. **Do you need strict audit, cost caps, and a single traceable control flow?** Lean orchestrator.
7. **Is the problem open-ended with no known plan?** Consider blackboard, magentic, or a single continuous-context agent [4][8].

**Key findings**

- Orchestrator = centralized top-down control (a lead agent decomposes, routes, and synthesizes); blackboard = decentralized shared-memory where a control unit dynamically picks agents and structure emerges during computation [1][2].
- Orchestrator wins for knowable-upfront work: routing/triage, independent parallel tool calls, fixed pipelines, and cases needing strict audit and predictable cost. Anthropic's lead-agent + isolated subagents beat single-agent Opus 4 by 90.2%, with token usage explaining 80% of variance [2][3].
- Blackboard wins for interdependent, emergent work: iterative refinement to consensus, cross-agent verification, constraint satisfaction. LbMAS beat CoT by 4.33% and static multi-agent by 5.02%; blackboard surveys report 13-57% gains over RAG/master-slave baselines [1][7].
- The decisive question is interdependence: isolated-worker orchestration fails when agents make conflicting implicit decisions (Cognition's Mario-vs-Flappy-Bird example); those tasks need shared context (blackboard) or a single continuous-context agent [8].
- Coding is generally a poor fit for parallel multi-agent orchestration because tasks are sequential, stateful, and interdependent; research/breadth-first exploration is the ideal orchestrator fit [2][8].
- 2026 consensus: use an orchestrator that OWNS context for decomposable work, keep dependent work in one continuous context, and cap collaborative blackboard/group-chat at ~3 agents to avoid infinite loops and coordination collapse [4].

## 7. Best practices & SOTA (2024–2026) + the bounded blackboard

### Two dominant coordination topologies

**Orchestrator-worker** and **blackboard** are the two production-relevant multi-agent topologies in 2024–2026. Orchestrator-worker (LangGraph, CrewAI, OpenAI Agents SDK, Anthropic's research system) uses a lead agent that plans, spawns specialized subagents in parallel, and synthesizes their results; centralized coordination gives "the best balance of simplicity, debuggability, and observability" [2]. The **blackboard** pattern instead uses a shared read/write state as the *exclusive* communication substrate, with a **control unit** selecting which agents act each cycle — solving the orchestrator "phone game" of lossy message passing [1][12]. Recent arXiv work shows blackboard systems (bMAS) match or exceed static and dynamically-orchestrated systems on reasoning/math while remaining token-efficient [1]; a data-science variant reports 13–57% relative gains in end-to-end success and up to 9% in discovery F1 by letting agents *self-select* against the board rather than requiring a coordinator that knows every agent's expertise [4].

### Bounding loops and the control unit

Every serious source treats **iteration caps as mandatory**. The bMAS paper bounds cycles to a hard maximum round limit (set to 4) OR early termination when a "decider" agent posts a solution — averaging only 2.88–3.29 rounds [1]. Anthropic embeds explicit scaling heuristics (simple=1 agent/3–10 tool calls; complex=10+ subagents) via prompt engineering to prevent failure modes like "spawning 50 subagents for simple queries" or "scouring the web endlessly" [8]. A deterministic control plane for coding agents enforces a **hard 3-iteration cap on auto-fix loops** inside a bounded sequential pipeline [10]. The control unit is the natural home for these bounds — but note a critical nuance for your "bounded blackboard" thesis: in the published bMAS the control unit is itself **LLM-based**, not deterministic [1]. Making that control unit deterministic (a state machine that enforces invariants and blocks on violation [10]) is exactly the gap the bounded/controlled-blackboard idea fills.

### Deterministic gates — "the LLM never renders the final verdict"

This is now an explicit 2025 industry principle. The pattern is **"LLM proposes, deterministic rules enforce, human authorizes changes to the rules"** [10]. The argument against LLM-as-gate is twofold: (1) a second LLM judging the first shares the same vulnerability class — "the same input that bends the agent bends the judge" — and (2) non-determinism makes audit impossible: "the same action gets waved through Tuesday and stopped Wednesday" [11]. The recommended architecture separates an LLM layer (propose, detect, flag) from a deterministic **enforcement layer** (hard rules that cannot be reasoned with) plus signed audit logs [11]. Related admission-control work (Agent Control Protocol, Organizational Control Layer, GuardAgent compiling policy to executable code) all place a non-probabilistic gate at the execution boundary [10]. MyAntFarm.ai reports multi-agent orchestration with **zero quality variance** across trials, enabling production SLAs impossible with stochastic single-agent output [10].

### Barrier-synced supersteps, reducers, state schema

LangGraph draws directly from Google's **Pregel**: computation runs in **supersteps** where nodes execute in parallel, then hit a **barrier sync** guaranteeing all messages sent in a superstep arrive at the start of the next [7]. State is a schema plus **reducer functions** that merge concurrent updates to the same channel deterministically (playing Pregel's Combiner role), so parallel branches don't clobber each other [7]. Best practice is to "define state, interrupts, and recovery first" rather than treating the graph as a longer chain [7]. This maps cleanly onto a bounded blackboard: the board is the channel set, reducers are the deterministic merge/gate, supersteps are the bounded cycles.

### Observability, WORM logs, and audit

**OpenTelemetry GenAI semantic conventions** are the emerging standard: a top-level `invoke_agent` span with child `chat` and `execute_tool` spans, and attributes `gen_ai.request.model`, `gen_ai.usage.input/output_tokens`, `gen_ai.response.finish_reasons`, plus captured `input.messages`/`output.messages` [3]. The design goal is "not just did it run, but what did it decide and why" [3]. Datadog, Honeycomb, New Relic, LangChain, CrewAI, AutoGen emit or ingest these [3]. **Langfuse** is a thin OTel-native layer (SDK v4) adding token/cost tracking, prompt linking, scoring, and **sessions** to group multi-turn traces [5]. For audit, EU AI Act, NIST AI RMF, and OWASP LLM Top 10 converge on **immutable, append-only / WORM logs with cryptographic hashing** so entries cannot be silently altered [6]. Recommended trace payload: input snapshot, retrieval sources, tool calls+params, outputs, validator results, cost, latency, correlation ID, approver, outcome label — tiered hot/warm/cold storage; overhead is ~5–10 ms/call [6].

### Guardrails, HITL gates, evaluation, cost

Guardrails: Llama Guard 4 (14 safety categories) and NeMo Guardrails (models whole conversation flows, not just I/O filtering) are standard, but a noted gap is that guardrails target *models, not agent action sequences* — ShieldAgent/GuardAgent address action-level policy compliance [9][10]. **HITL gates**: LangSmith annotation queues route runs to experts, and eval thresholds fail CI pipelines automatically (pytest/Vitest/GitHub) [x]. **Evaluation**: trajectory evals score the exact sequence of messages/tool-calls via match or LLM-judge (agentevals); start with ~20 queries and LLM-as-judge rubrics (factual/citation accuracy, completeness, source quality, tool efficiency) [8][x]. **Cost**: multi-agent uses ~15× chat tokens, and token usage explains ~80% of performance variance — so context isolation (fresh windows per subagent) is the primary cost lever, plus strict context ceilings tracked before hitting API limits [8][8b].

### Verdict on the bounded/controlled blackboard

The term is **not yet a formalized named pattern** in literature — searches for "bounded/controlled blackboard" return only the generic blackboard pattern [1][12]. But every ingredient is validated independently in 2025 work: the blackboard substrate + control unit [1][4], hard iteration caps [1][10], deterministic gates where the LLM never renders the verdict [10][11], and barrier-synced deterministic reducers [7]. The novel contribution of a "bounded/controlled blackboard" is fusing them: replacing bMAS's *LLM* control unit with a *deterministic* control unit + gate over a reactive shared state. That is a defensible, under-explored design.

**Key findings**

- Blackboard LLM systems (bMAS) use a control unit + shared board as the sole comms substrate and can match/exceed orchestrator systems while staying token-efficient; but in the published bMAS the control unit is LLM-based, not deterministic [1][4].
- Hard iteration caps are universal best practice: bMAS caps at 4 rounds (avg ~3) [1]; deterministic coding-agent control planes enforce a 3-iteration auto-fix cap [10]; Anthropic bounds subagent count/tool-calls via prompt heuristics [8].
- 2025 industry principle: 'LLM proposes, deterministic rules enforce, human authorizes rule changes' — an LLM must never be the final gate because a judge-LLM shares the agent's vulnerability class and non-determinism makes audit impossible [10][11].
- LangGraph = Pregel: bounded supersteps with barrier sync, and deterministic reducer functions merge concurrent state updates so parallel branches don't clobber the shared channel [7].
- OpenTelemetry GenAI semantic conventions (invoke_agent/chat/execute_tool spans, gen_ai.usage.*, finish_reasons) are the emerging standard; Langfuse is an OTel-native layer adding cost/session tracking [3][5].
- Audit requires immutable append-only/WORM logs with cryptographic hashing to satisfy EU AI Act / NIST AI RMF / OWASP LLM Top 10; overhead ~5-10ms/call [6].
- Guardrails (Llama Guard 4, NeMo Guardrails) target models not agent action sequences — a gap filled by action-level enforcers like ShieldAgent/GuardAgent that compile policy to deterministic code [9][10].
- Multi-agent systems use ~15x chat tokens and token usage explains ~80% of performance variance, so per-subagent context isolation with strict ceilings is the main cost lever [8].
- Evaluation uses trajectory evals (exact message/tool-call sequence via match or LLM-judge) plus HITL annotation queues and CI eval-threshold gates [8][x].
- 'Bounded/controlled blackboard' is NOT yet a named pattern in the literature, but all its components are independently validated in 2025 work — fusing a DETERMINISTIC control unit + gate onto a reactive blackboard is a genuine, under-explored design.

## 8. Hybrids, third topologies & the bounded blackboard

### Hybrid patterns and the bounded/controlled blackboard

The orchestrator/blackboard split is a spectrum, not a binary, and the most robust production systems live in the middle.

**A third and fourth topology round out the map.** Beyond the two poles sit *moderated group-chat / debate / society-of-mind* (AutoGen's `GroupChatManager`, `RoundRobinGroupChat`, `SelectorGroupChat`; multi-agent debate à la Du et al.; CAMEL role-play as the two-agent limit) — a single shared *conversation transcript* where a thin moderator only selects speakers and terminates, holding no plan and doing no synthesis. It is neither orchestrator (no decomposition/delegation) nor blackboard (no structured shared workspace), and it carries topology-specific pathologies: token blowup from rounds×agents, and sycophancy/conformity where homogeneous debate behaves like a martingale that cannot beat majority vote in expectation. And a *learned/searched orchestration* frontier (ADAS/Meta Agent Search over a code search space, AFlow's MCTS over code workflows, GPTSwarm's graph-optimized swarms, MaAS's query-adaptive agentic supernet) reframes the whole question as a *search over coordination structures* — both orchestrators and blackboards are points in that space — though a 2025 critique ("Inefficiencies of Meta Agents") shows the design-cost economics rarely close.

**Hybrid patterns in practice.** Real deployments routinely blend: a supervisor that *owns* a shared task list its workers can all read (Anthropic's "Agent Teams" direction, adding peer messaging and file locking to defeat the no-mid-task-coordination limit); Bedrock's "supervisor with routing" mode that bypasses full orchestration for simple requests; a blackboard whose control unit is itself an orchestrator; or the 2026 convergence where Cognition — after arguing "Don't Build Multi-Agents" — shipped a coordinator over isolated managed Devins. A persistent-memory layer (Mem0 pipelines, Zep's bi-temporal knowledge graph with fact invalidation, Letta's OS-style RAM/disk hierarchy with sleep-time consolidation, A-MEM's self-organizing notes) is increasingly the substrate *beneath* any shared state — Letta's shared memory blocks are a literal database-backed blackboard, and they surface exactly the write-arbitration hazards (last-writer-wins) the classic blackboard control shell existed to solve.

**The bounded/controlled blackboard, positioned neutrally.** This is a reactive shared state governed by a *deterministic* control unit and gate. It is not (yet) a named pattern in the literature — searches return only the generic blackboard — but every ingredient is independently validated in 2025 work, and fusing them is a defensible, under-explored design rather than a proven winner:

- **Reactive shared board** as the sole communication/memory substrate (bMAS, PatchBoard), giving mid-flight re-triggering and WORM auditability.
- **A deterministic control unit** replacing the *LLM-based* control unit used in the published bMAS — a state machine that enforces invariants, selects only agents whose preconditions changed, and blocks on violation.
- **A deterministic gate** where the LLM proposes and non-probabilistic rules enforce, because an LLM judge shares the agent's vulnerability class and non-determinism makes audit impossible.
- **Barrier-synced, bounded supersteps with reducer functions** (LangGraph's Pregel/BSP model): the board is the channel set, reducers are the deterministic merge, supersteps are the bounded cycles, and a hard round cap guarantees termination.
- **Schema-grounded, atomic, role-scoped, per-agent-logged mutations** validated at the write boundary (PatchBoard-style JSON-Patch over a transactional kernel).

Framed honestly: the bounded blackboard trades some of the pure blackboard's emergent flexibility for the orchestrator's reproducibility, cost bounds, and audit story. It is a *design position*, not a benchmarked verdict — its value is that it isolates exactly which orchestrator guarantees you can graft onto a reactive board, and at what cost to adaptivity.

**A concurrency-theory footnote worth stating explicitly:** CRDTs (as used by CodeCRDT) provide *strong eventual consistency* — replicas converge — but **not** semantic/application-level consistency, which is precisely why residual semantic conflicts (duplicate declarations, type mismatches) persist even when merges never fail. That is the formal reason a deterministic gate/validator at the write boundary, not just a CRDT merge, is needed for a truly disciplined board.

## 9. Common pitfalls & failure modes

### Common pitfalls and how the patterns fail

**Orchestrator-specific traps.**
- **Vague delegation.** Underspecified worker prompts caused duplicate work and over-spawning — Anthropic saw 50 subagents spawned for a trivial query. Give each worker an explicit objective, output format, tools, and boundaries, plus scaling heuristics (simple = 1 agent / 3–10 tool calls; complex = 10+).
- **The hub as bottleneck and SPOF.** A serialized hub call rate-limits the whole system: if the decomposition call is serial and takes ~3s, the hub emits roughly one decision per ~3s (~0.33 tasks/sec) regardless of how many workers idle behind it — *that* low ceiling is what "bottleneck" means. (A widely copied "~6.7 tasks/sec" figure circulating in SEO blogs is a dimensionally invalid workers÷latency division and should not be cited as a measured ceiling.)
- **Context crowding.** Routing accuracy degrades after ~8–12 subagent round-trips as accumulated history crowds out current state; the orchestrator can blow past context limits at 4+ workers.
- **The message-restatement tax.** Because workers can't answer the user directly, the supervisor must translate/restate their output — the measured reason a swarm sometimes beats a supervisor on both tokens and latency.

**Blackboard-specific traps.**
- **Write contention and races.** Concurrent unstructured writes let one agent's update clobber another's; last-writer-wins semantics (as in Letta's `memory_rethink`) are unsafe for simultaneous multi-agent edits — hence single-owner-write patterns and schema-validated, role-scoped mutations.
- **Semantic conflicts survive convergence.** Even with CRDT convergence and zero *merge* failures, CodeCRDT measured a ~5–10% *semantic* conflict rate (verified against arXiv 2510.18893; the paper reports up to 21.1% speedup on some tasks, up to 39.4% slowdown on others, and 100% convergence across 600 trials — its measured sweet spot is 3–5 agents, with larger-N degradation labeled *projected*, not observed).
- **Nondeterminism and "no single trace."** An ad-hoc board lacks convergence guarantees and can be hard to replay; discipline (versioned WORM writes) is what converts it into an auditable asset.
- **Debuggability.** Coordination-type failures dominate multi-agent systems generally; blackboards make them harder to localize because the failure is in the *interaction*, not one agent.

**Cross-cutting traps (both / all topologies).**
- **Token blindness.** Multi-agent uses ~15× chat tokens; the primary cost lever is per-subagent context isolation with strict ceilings tracked *before* hitting API limits.
- **Reasoning-model economics are lumpier than "cheaper every year."** Thinking tokens bill at the output rate; a ~32% "price reversal" means many nominally cheaper reasoning models cost more per task once hidden reasoning tokens are counted — and an "overthinking tax" (5–10× longer outputs even when correct) means budgets must be *adaptive*, not maxed.
- **No compute normalization.** Many multi-agent "wins" are confounded by simply spending more test-time compute; always compare under equal token budgets.
- **Conformity in debate.** Diversity is not a free fix — heterogeneous model teams underperformed the best homogeneous member in 6 of 8 tested condition-task pairs; use anonymization, anti-conformity prompts, and score-based aggregation.
- **Security blast radius scales with topology.** An orchestrator on MCP concentrates control *and* tool-poisoning risk in one hub (e.g. CVE-2025-54136; MCP mandates no client-side metadata validation). A blackboard/mesh distributes control but multiplies pairwise trust boundaries and enables *injection amplification*: a single poisoned board entry can re-trigger many agents. Shared-state contagion and confused-deputy-via-handoff are genuine multi-agent-specific attack surfaces that single-agent testing cannot surface.
- **Thin benchmark ground.** Don't rest an architecture claim on one blog plus one internal eval. Anchor to public, cost-aware yardsticks — GAIA, AppWorld, WebArena/VisualWebArena, SWE-bench Verified/Multimodal, OSWorld, BrowseComp, with HAL as the cost-normalized meta-leaderboard (whose finding that more reasoning budget *lowered* accuracy in 21 of 36 settings is a caution against literal headline numbers).

## 10. How to showcase it

### Recommended demo repo: one task, both patterns, full instrumentation

**Thesis of the repo.** No existing repo cleanly pits orchestrator vs. blackboard on one instrumented task — that gap is the opportunity. The design principle, borrowed from LangChain's benchmarking study, is **hold the task constant, sweep one complexity knob, and let the curves diverge.** The separating axis is *constraint interdependence*: orchestrators win at low coupling (routing-like, parallelizable), blackboards win at high coupling (tightly-linked cross-checks).

#### Repo structure
```

├── task/

│   ├── constraints.py            # coupling knob (0=independent → high=rippling cross-checks)
│   └── tools.py                  # flights, rooms, catering, calendar — SAME tools for both
├── shared/
│   ├── llm.py                    # single pinned model + adaptive thinking-budget flag

│   └── worm_log.py               # append-only event-sourced JSON log (blackboard writes)
├── orchestrator/
│   ├── supervisor.py             # LangGraph supervisor: classify → dispatch → synthesize
│   └── workers.py                # flight/room/catering specialists, isolated contexts
├── blackboard/
│   ├── board.py                  # StateGraph shared state; reducers per channel
│   ├── control_unit.py           # deterministic: re-trigger only changed-precondition agents
│   ├── gate.py                   # schema-validated, atomic, role-scoped write gate
```

#### Demo task (one task, one knob)
**Multi-constraint meeting/travel scheduling** — schedule N people across timezones under room capacity, budget, calendar conflicts, and dietary/accessibility constraints, where satisfying one constraint (a cheaper flight) *invalidates* another (a booked room), forcing re-checks. This is the class Terrarium already ships (worth forking as the blackboard scaffold) and PatchBoard-style validated state fits cleanly. Expose exactly one knob — **constraint interdependence** (0 = each attendee independent → routing-like; high = tightly coupled ripples). Same task, same tools, same pinned LLM for both runs — only topology and the knob change. The crossover point where blackboard overtakes orchestrator *is* the punchline.

- **Orchestrator run:** supervisor classifies subtasks (flights/rooms/catering), dispatches specialists in isolated contexts, collects returns, re-plans. Efficient and latency-winning at low coupling; thrashes on serialized re-planning as coupling rises.
- **Blackboard run:** specialists write partial solutions/constraints to the shared board; the deterministic control unit re-triggers *only* agents whose preconditions changed (room agent re-fires when budget agent lowers the cap). Converges via targeted re-triggers at high coupling.
- **Bounded-blackboard variant (optional third lane):** the same board with a hard superstep cap and a deterministic write gate — demonstrates the hybrid's bounded cost and clean audit trail against the pure blackboard.

#### Instrumentation (the load-bearing part)
Emit **OpenTelemetry GenAI semantic-convention spans** so both topologies are measured identically: a top-level `invoke_agent` span with child `chat` and `execute_tool` spans, carrying `gen_ai.request.model`, `gen_ai.usage.input/output_tokens`, `gen_ai.response.finish_reasons`, latency, and tool name — **plus custom `message.count` and `re_trigger.count` attributes**. Cost = tokens × price. Log every blackboard write as an **append-only/WORM event** (event-sourcing) so runs are replayable and diffable — the auditability "aha" the orchestrator's transient messages lack. Trace decision patterns/interaction structure, not payload contents (privacy posture). **Report reasoning-token spend separately** and run at least one sweep with an adaptive thinking budget, so the repo also demonstrates the single-reasoning-agent-vs-fan-out substitution effect and normalizes comparisons by compute.

**Headline metrics per run, plotted against the coupling knob:** total tokens (with reasoning tokens broken out), wall-clock latency, message/turn count, agent-invocation count, re-trigger count, task success. Expect orchestrator tokens/latency to climb steeply with coupling while the blackboard stays flatter — the LangChain-shaped curves, inverted in the blackboard's favor past the crossover.

#### Visualization design
1. **Live agent-interaction graph.** Orchestrator: animated supervisor↔specialist *message-flow* edges. Blackboard: *writes into a central board node* plus *re-trigger arrows* back out. LangGraph Studio renders this for free on LangGraph graphs; Arize Phoenix (OpenInference/OTel-native, open-source) and Langfuse's agent-graph view both ingest the OTel spans above.
2. **Trace + WORM-log viewer.** A waterfall trace shows parallel subagent spans as concurrent bars for the orchestrator vs. a serialized-then-fanned re-trigger pattern for the blackboard. Pair with an append-only log viewer rendering each mutation as a timestamped, replayable event.
3. **Side-by-side counter panel + crossover chart.** Live token/latency/message counters per architecture and the divergence plot that ties the story together.

**Recommended stack:** LangGraph + LangGraph Studio for graphs; OpenTelemetry GenAI semconv for uniform instrumentation; Arize Phoenix or Langfuse (both open-source, OTel-native) for traces; a lightweight event-sourced JSON log for the WORM viewer; the dashboard served via `.claude/launch.json` for live preview. Optionally validate the demo's own claims against a public yardstick (e.g. an AppWorld- or scheduling-style multi-app slice) so the repo's numbers sit on reproducible ground rather than a bespoke eval alone.

### 10.1 Survey of demo approaches

### The comparison landscape today

There is no single canonical repo that puts an **orchestrator (supervisor)** and a **blackboard** architecture head-to-head on one task with full instrumentation — which is exactly the gap a good demo fills. But the pieces exist and point clearly at what "compelling" looks like.

The most instructive prior art is LangChain's own **"Benchmarking Multi-Agent Architectures"** study [1]. It runs the *same* task set (a modified τ-bench extended with 6 distractor domains, each with ~19 tools) through three topologies — single-agent, swarm (peer hand-offs), and supervisor (central delegator) — and measures **accuracy and token usage** as complexity scales. Its findings are the template for a fair demo: single-agent accuracy "falls off sharply" with ≥2 distractor domains and token use grows, while supervisor and swarm stay flat on cost; the supervisor pays a "translation" penalty because sub-agents can't answer users directly [1]. This proves the design principle: **hold the task constant, sweep a complexity knob, and let the curves diverge.**

On the blackboard side, 2025 produced strong reference designs. The **LbMAS blackboard system** [2][3] replaces a coordinator with a shared board: the main agent *posts a request*, helper agents *autonomously decide whether to contribute*, and a **control unit re-evaluates the board each cycle** to select which agents fire next — iterating until a decider agent confirms a solution. A companion paper [4] frames the same control loop over MMLU/GPQA/MATH and reports +5% over static orchestration, crediting "dynamic adaptation of collaboration" versus "pre-defined agent roles." **PatchBoard** [5] sharpens the auditability angle: agents emit *validated JSON-Patch mutations* over shared structured state, a deterministic kernel commits them transactionally and **records replayable logs** — 84.6% success at 45.5k tokens/task on ALFWorld. **Terrarium** [6] (code at github.com/umass-aisec/Terrarium) is the closest to a ready testbed: a *modular, configurable* blackboard harness with per-message logging and observability, shipping DCOP tasks like **meeting scheduling** and smart-home energy coordination.

### When each architecture wins — the design axis

The literature converges on one axis. Anthropic's orchestrator-worker research system [7] wins on **heavy parallelization, breadth-first exploration, and independent sub-directions** — it beat single-agent by 90.2% on research, but explicitly *fails* on "tasks requiring shared context across all agents or tight dependencies between components." The blackboard papers claim the mirror image: they shine when **problem structure emerges during computation** and sub-results are **interdependent** [2][4]. So: **routing/classify-and-dispatch → orchestrator; interdependent multi-constraint cross-check → blackboard.**

### Recommended demo task (one task, one knob)

Use **meeting/travel scheduling under interacting constraints** — the exact class Terrarium already ships [6] and PatchBoard-style state fits cleanly [5]. Concretely: schedule N people across timezones with room capacity, budget, calendar conflicts, and dietary/accessibility constraints, where satisfying one constraint (e.g., a cheaper flight) *invalidates* another (a booked room), forcing re-checks.

Expose one knob — **constraint interdependence** (0 = each attendee independent → routing-like; high = tightly coupled constraints that ripple).

- **Orchestrator run:** a supervisor classifies sub-tasks (flights, rooms, catering) and dispatches specialists; specialists return to the supervisor, which re-plans. At low coupling this is efficient and wins on latency/tokens.
- **Blackboard run:** specialists write partial solutions/constraints to a shared board; the control unit **re-triggers** any agent whose preconditions changed (a room agent re-fires when the budget agent lowers the cap). At high coupling the orchestrator thrashes on serialized re-planning while the blackboard converges via targeted re-triggers.

This is fair because it's the **same task, same tools, same LLM** — only topology and one knob change, mirroring [1]. It cleanly separates the two: the crossover point where blackboard overtakes orchestrator *is* the demo's punchline.

### Instrumentation (the load-bearing part)

Emit **OpenTelemetry GenAI spans** so both topologies are measured identically [8][9]: one span per LLM call and per tool call, with standard attributes for prompt/completion **tokens, latency, model, tool name**, plus a custom **message-count** and **re-trigger-count** attribute [8]. Cost = tokens × price [9]. Anthropic's practice of tracing "decision patterns and interaction structures" without reading contents [7] is the right privacy posture. Log every blackboard write as an **append-only / WORM event** (event-sourcing) so runs are replayable and diffable [10][5].

Headline metrics per run, plotted against the coupling knob: **total tokens, wall-clock latency, message/turn count, agent-invocation count, re-trigger count, task success**. Expect orchestrator tokens/latency to climb steeply with coupling while blackboard stays flatter (the [1] shape, inverted for the blackboard's favor).

### Visualization tooling

Two complementary views make the contrast legible:

1. **Live agent-interaction graph.** For the orchestrator, render the supervisor↔specialist **message flow** as animated edges; for the blackboard, render **writes into a central node** plus **re-trigger arrows** back out. Recommended stack: **LangGraph + LangGraph Studio** for graph rendering [11], **OpenTelemetry GenAI semconv** for uniform instrumentation [8][9], **Arize Phoenix or Langfuse** (both open-source, OTel-native) for traces and agent graphs [12], and a lightweight event-sourced JSON log for the WORM viewer [10][5]. ## 11. Frontier & additional findings

### The Protocol Substrate: MCP and A2A as the Wiring Layer for Orchestrator vs Blackboard Topologies

### Why the dossier's topology discussion needs a protocol layer

Orchestrator-vs-blackboard is a *logical* control-topology distinction; by 2026 it is almost always *physically realized* over one of two open protocols. Anthropic's **Model Context Protocol (MCP)** and Google's **Agent2Agent (A2A)** protocol are now the interoperability substrate on which these topologies are wired, and they are explicitly positioned as complementary layers rather than competitors: MCP is the *vertical* agent-to-tool/resource layer, A2A the *horizontal* agent-to-agent layer [1][6][8]. Google states directly that "A2A is an open protocol that complements Anthropic's Model Context Protocol (MCP), which provides helpful tools and context to agents" [6]. Treating topology in a protocol vacuum misses that the protocol *itself* biases you toward a control shape.

### MCP: the client-server primitive that underwrites hub-and-spoke control

MCP is JSON-RPC 2.0 over a duplex transport (stdio or Streamable HTTP with optional SSE), with stateful sessions and capability negotiation [1][11]. Its architecture is intrinsically **hub-and-spoke**: a *host* (the LLM app) runs one *client* per *server*, and each client connects to exactly one server [11]. Servers expose three primitives — **tools** (model-controlled), **resources** (app-controlled), and **prompts** (user-controlled); clients expose **sampling**, **roots**, and (since the 2025-06-18 spec) **elicitation** [11]. The 2025-06-18 revision also added **tool output schemas** (structured outputs) and removed JSON-RPC batching [11][12]. Critically, MCP has *no* notion of a peer agent — the server is a subordinate capability provider. This makes MCP the natural wire for an **orchestrator**: a supervisor agent holds the control loop, and everything it commands (tools, data, even sub-models via `sampling`) hangs off it as a spoke. A survey of interoperability protocols classifies MCP's topology explicitly as "Client-Server" hub-and-spoke with "manual registration or static URL lookup" discovery [2].

### A2A: peer messaging that can wire either topology

A2A reuses the same transport family — HTTP(S), SSE, JSON-RPC 2.0 — but adds gRPC and HTTP+REST bindings mapping to one canonical Protobuf data model [3][5]. Its defining primitives are the **Agent Card** (a JSON capability manifest served at `/.well-known/agent-card.json`), the stateful **Task** (lifecycle: `submitted → working → input-required → completed/failed/canceled/rejected`), **Message**/**Part**, and **Artifact** outputs [4][5]. Its core method is `message/send`; updates arrive via polling, SSE streaming, or webhook push notifications [4][5]. The pivotal design principle is the **opaque agent**: agents "collaborate without needing access to each other's internal state, memory, or tools" and interact "in their natural, unstructured modalities, even when they don't share memory, tools and context" [5][6]. A2A distinguishes a **client agent** (formulates tasks) from a **remote agent** (acts on them) [6].

That client/remote asymmetry is why A2A can wire *both* topologies. Wire a single supervisor as the client agent delegating to specialist remote agents and you have an **orchestrator/supervisor topology** — exactly the pattern LangGraph, Google's ADK, CrewAI and the OpenAI SDK implement, where "the first role is the hub (coordinator)… the hub receives the user goal and a framework-native `delegate_to_agent` tool" [7]. ADK lets an agent "delegate a task to a LangGraph or CrewAI agent without framework-specific integrations" [7]. Conversely, let arbitrary agents discover each other via Agent Cards and message peer-to-peer, and A2A approximates a decentralized mesh — the survey labels A2A's native topology "Peer-like Client ↔ Remote Agent" with "inter-agent negotiation," contrasted with MCP's hub-and-spoke [2].

### A2A does *not* natively give you a blackboard

This is the load-bearing point for the dossier. A2A is **point-to-point message passing**, not shared state. It "provides no mechanism for coordinating when and how agents should interact" and "presumes pre-defined roles, static discovery, and explicit coordination… does not provide a framework for spontaneous collaboration, distributed consensus, or emergent task sharing" [9][14]. At enterprise scale the topology cost is explicit: the HiveMQ analysis names an **"N-squared connectivity problem"** — 4 agents need 6 connections, 50 agents need ~1,200 — plus tight coupling (each agent must know every peer's endpoint/auth/availability) and cascading failure, because "a single agent's downtime… can cascade into system-wide failures" [10]. Its recommended fix is precisely a **blackboard substrate**: replace point-to-point calls "with a central message broker" (pub/sub, MQTT) where "agents post their outputs and consume relevant published data," inverting the mesh into a shared event backbone and enabling dynamic workflow composition [10]. In other words, a true blackboard topology is not A2A "out of the box" — it is A2A (or MCP) agents plus a shared-state/message-broker layer bolted on top, or a purpose-built pub/sub fabric like AGNTCY's SLIM (Secure Low-latency Interactive Messaging) [8].

### The emerging stack and its security implications

The 2026 picture is a **layered protocol stack**: MCP for tool/context, A2A for agent collaboration, ACP (IBM/Linux Foundation) for runtime-neutral REST messaging, ANP for open-internet DID-based discovery, and AGNTCY/OASF for directory-level discovery, unified in academic proposals as a single "Agent Communication Layer" [2][8][13]. A2A was donated by Google to the **Linux Foundation in June 2025** (announced June 23–25), growing to 150+ supporting organizations and remaining vendor-neutral [15]. Transport has converged: "HTTPS with SSE for streaming has won… MCP HTTP+SSE, A2A's Streamable HTTP, and ACP's REST+SSE all land at the same shape" [8].

The protocol choice also determines the **attack surface** of your control topology. MCP's hub concentrates risk in tool descriptions: **tool poisoning** (CVE-2025-54136) exploits that "the MCP specification does not require client-side validation of server-provided metadata," and the MCPTox benchmark found live servers broadly vulnerable [16]. A2A's peer mesh raises the open question of *contagion* — "can compromised agents 'infect' others through A2A?" — and the opaque-agent principle is in tension with the fact that agents "must often expose internal logic, memory, or tool access to work together" [9][16]. For the dossier: an orchestrator topology on MCP centralizes both control *and* the poisoning blast radius; a blackboard/mesh topology on A2A distributes control but multiplies pairwise trust boundaries and lacks a native coordination governor — which is exactly why real deployments add a broker or supervisor layer back on top.

### Group-Chat / Debate / Society-of-Mind: The Moderated-Conversation Topology as a Distinct Third Family

### The third topology: moderated group conversation

The orchestrator-vs-blackboard axis misses a large, distinctly-shaped family that dominated 2023–2026 collaborative-agent research and tooling: agents talking to each other in a **shared, single-threaded conversation** where a lightweight moderator decides who speaks next. This is neither a pure orchestrator (no agent decomposes a plan and delegates typed subtasks to workers who report back) nor a pure blackboard (there is no structured, indirectly-read shared workspace accumulating hypotheses and partial results). Instead, all participants subscribe and publish to the *same message topic*, and every utterance is broadcast to everyone, so coordination happens through the transcript itself rather than through a delegation tree or a data structure [6][10].

**Group chat (AutoGen / AG2).** AutoGen popularized the pattern. A `GroupChatManager` runs a three-step loop: dynamically select the next speaker, collect that speaker's response, and broadcast it to all participants who share one context thread [1][6]. Selection can be `auto` (an LLM picks the next speaker from participant `name`/`description` metadata plus conversation history), `round_robin`, `random`, `manual`, or a custom function/FSM [1][2]. In the v0.4/AgentChat rewrite (stable since Jan 2025) this crystallized into two presets: `RoundRobinGroupChat` (sequential turn-taking) and `SelectorGroupChat` (model-based next-speaker selection that by default won't pick the same speaker twice unless `allow_repeated_speaker=True`), both covering v0.2's `GroupChat` capabilities [4][10]. Crucially, the core docs describe the manager as a *sequential orchestrator maintaining turn-taking discipline* that "doesn't directly transfer context between agents — it broadcasts to a shared space," explicitly contrasting it with **handoff** patterns where control transfers bilaterally [6]. That is the signature difference from orchestrator topologies: shared collective awareness vs. point-to-point delegation. AG2's v0.9 (Apr 2025) reworked this into a dedicated "Group Chat Pattern" API, signaling continued centrality [7].

**Multi-agent debate (Du et al., 2023 → ICML 2024).** Here the moderation is even thinner. N instances of the same model independently answer, then each is shown peers' answers and asked to critique and revise across R rounds; the final answer is aggregated (typically majority/plurality). The canonical config was 3 agents × 2 rounds "due to computational cost," with gains scaling in both dimensions, improving math/strategic reasoning and factual validity while cutting hallucinations across six benchmarks [3][8]. Du et al. explicitly frame this as a "multiagent society" needing only black-box model access [8] — the intellectual lineage to Minsky's *Society of Mind*, where intelligence emerges from many simple interacting agents, is made directly in the follow-on literature [7][9].

**CAMEL role-play (Li et al., NeurIPS 2023).** A degenerate but influential two-agent case: an AI-User and AI-Assistant are seeded by a task-specifier and driven by "inception prompting" to autonomously cooperate until task completion, explicitly to *study* the behavior of a "society of agents" and generate conversational data [5]. It shares group-chat's DNA (role-conditioned agents conversing to a goal) without a separate moderator.

### Why it's genuinely a separate topology

The distinguishing invariants: (1) a **single linear transcript is the coordination medium** — not a plan tree, not a typed shared store; (2) coordination is via **turn-taking and broadcast**, so context is redundantly shared with all agents every turn; (3) the moderator, when present, does **only speaker selection and termination**, holding no task plan and doing no result synthesis — unlike an orchestrator that owns global state and composes worker outputs [6][11]. A communication-centric survey characterizes this "joint-chat scheme" (all agents on a common channel enabling rapid propagation but demanding turn-taking/conflict-resolution mechanisms) as categorically distinct from blackboard architectures, where agents interact *indirectly* through a structured workspace of accumulated intermediate hypotheses rather than a conversation [12]. One pointed critique notes AutoGen group chat "isn't a true group chat because the agents don't have agency — a Moderator dynamically selects who talks next" [12], which is exactly why it sits between orchestrator (moderator has agency, agents don't) and swarm (agents have agency, no moderator).

Positioning against the dossier's existing poles: **vs. orchestrator** — no decomposition/delegation/synthesis; the "manager" is a router, not a planner [6][11]. **vs. blackboard** — no read/write shared data structure; the transcript is append-only prose broadcast in full, and blackboard papers themselves draw this contrast [12][and blackboard-architecture line of work]. **vs. swarm** — swarm eliminates central control entirely (autonomous peers acting on shared state/environment signals), whereas moderated group chat retains a centralized turn-allocator [11][12].

### Practical trade-offs and 2024–2026 critique

Deployment reality tempers the "most-deployed collaborative pattern" claim: one industry breakdown estimates two-agent assistant+proxy setups cover ~60% of AutoGen production use while group chat is under ~15%, with guidance to "use group chat only when you actually need rotating roles" [13]. It is arguably the most-*studied* collaborative pattern and the default multi-agent primitive in the most-downloaded framework, but simpler two-agent chains win in production.

The research critique is now substantial and belongs in the dossier: (a) **cost** — rounds × agents multiplies tokens; sparsifying communication to helpful agent pairs cuts token cost up to ~94.5% within 2% accuracy [15]. (b) **conformity/sycophancy** — RLHF-aligned agents discard correct latent reasoning to adopt the modal peer answer; homogeneous debate behaves like a martingale over beliefs and provably *cannot exceed majority vote in expectation* [15][16]. (c) **cost-effectiveness** — for 7–8B models, isolated self-correction beats unguided homogeneous debate on cost-accuracy [15]. Mitigations: heterogeneous model families (not just prompt-role variation) for decorrelated errors, response anonymization, anti-conformity prompts, and score-based aggregation — though heterogeneous teams underperform the best homogeneous member in 6/8 tested conditions, so diversity is not a free win [16]. These failure modes (non-convergence, false consensus, "Microsoft recommends ≤3 agents") are specific to the shared-transcript topology and don't map cleanly onto orchestrator or blackboard analyses [11][16].

**Recommendation for the dossier:** treat moderated group-chat/debate as a first-class third topology with its own axis label (e.g., "conversational / turn-taking broadcast"), positioned between centralized orchestration and decentralized swarm, with CAMEL/role-play as its two-agent limit and debate as its aggregation-focused variant.

### The Learned/Searched Orchestration Frontier: AFlow, ADAS, GPTSwarm, and Agentic Supernets

### Learned and Searched Orchestration: A Third Design Axis

The orchestrator-vs-blackboard framing captures two *hand-built* coordination paradigms, but a third line of work reframes the question entirely: rather than a human choosing and wiring a topology, an outer optimization loop **searches** for the coordination structure itself. This "automated agentic workflow" frontier directly competes with both hand-built orchestrators and blackboards, and treating AFlow as merely a token-cost baseline understates its significance. The core insight, articulated most sharply by ADAS, is a bet that "hand-designed solutions are eventually replaced by learned solutions," analogous to how Neural Architecture Search displaced hand-tuned CNN features [3].

**ADAS / Meta Agent Search (Hu, Lu, Clune, 2024; ICLR 2025).** ADAS defines the search space as *code itself*—a Turing-complete space in which an agentic system is an arbitrary program, so any orchestrator, debate, or blackboard-like pattern is in principle representable [3]. Its algorithm, Meta Agent Search, uses a "meta agent" that iteratively programs new agents, evaluates them, appends them to a growing **archive**, and conditions the next proposal on that archive (a stepping-stone / open-ended search) [2][3]. Reported gains over state-of-the-art hand-designed agents are substantial: +13.6 F1 on DROP, +14.4% on MGSM, and, after *transferring across domains*, +25.9% on GSM8K and +13.2% on GSM-Hard—with agents retaining superiority when moved across domains and models [3]. Critically for a security-conscious dossier, ADAS ships an explicit warning that it "involves executing untrusted model-generated code," which can act destructively—an inherent hazard of the code-as-search-space design [4].

**AFlow (Zhang et al., 2024; ICLR 2025 Oral).** AFlow keeps the code representation but replaces ADAS's "linear heuristic search" with a **Monte Carlo Tree Search** variant, arguing ADAS "struggles to generate effective workflows within a limited number of iterations" [1]. Its key efficiency lever is a set of reusable **Operators**—Generate, Format, Review, Revise, Ensemble, Test, Programmer, plus a Custom default—that encapsulate common agentic patterns (e.g., Self-Consistency Ensemble, Review-and-Revise) so the search need not rediscover them node by node [1]. Evaluated on six benchmarks (HumanEval, MBPP, MATH, GSM8K, HotpotQA, DROP), AFlow beats manually designed methods (IO, CoT, CoT-SC, MultiPersona Debate, Self-Refine, MedPrompt) by **5.7% on average** and surpasses prior *automated* methods (including ADAS) by **19.5%**, reaching 80.3% average [1]. The widely cited cost figure—matching GPT-4o "at 4.55% of its inference cost"—comes from letting AFlow-optimized *smaller* models (DeepSeek-V2.5, GPT-4o-mini) execute workflows that a Claude-3.5-Sonnet optimizer discovered [1]. This is the number the dossier already cites, but the surrounding claim is that learned workflows push a Pareto frontier where cheap models beat expensive ones.

**GPTSwarm (Zhuge et al., 2024; ICML 2024 Oral).** GPTSwarm sits between the two: it represents a swarm of agents as an **optimizable computational graph** where nodes are LLM/tool operations and edges are information flow, then optimizes at two levels—*node optimization* (refining per-node prompts, OPRO-style) and *edge optimization* (learning graph connectivity via a REINFORCE estimator over real-valued edge parameters) [5]. Reported results: Mini-Crosswords 0.80 (vs 0.675 Tree-of-Thought), HumanEval 0.88 (vs 0.76 unoptimized), and GAIA roughly doubling a GPT-4-Turbo baseline [5]. Two findings matter for the dossier's argument. First, "the quality of connections, rather than their quantity," drives performance—an empirical case that *topology*, not just agent count, is the lever both orchestrators and blackboards are implicitly tuning. Second, edge optimization can act as a **robustness mechanism**: in an adversarial setting, the optimizer learned to *disconnect* adversarial agents, recovering baseline performance—an automated analogue to trust/gating a blackboard would need hand-coded [5]. AFlow notes GPTSwarm's graph representation "struggles to represent workflows with conditional states," which is precisely why AFlow and ADAS moved to code [1].

**2025 frontier — query-adaptive supernets (MaAS).** The newest direction attacks a shared weakness of AFlow/ADAS/GPTSwarm: they all output a *single, static, one-size-fits-all* system that misallocates compute regardless of query difficulty. MaAS (Zhang et al., 2025) instead learns an **agentic supernet**—"a probabilistic and continuous distribution of agentic architectures"—and samples a *query-dependent* sub-workflow per input, tailoring LLM calls, tool use, and tokens to difficulty [6]. It reports needing only 6–45% of competitors' inference cost while outperforming them by 0.54–11.82%, with cross-dataset and cross-backbone transfer [6]. This is conceptually a blackboard-like "route to the right specialists on demand," but learned rather than hand-wired.

**The critical counter-evidence (important for a balanced SOTA claim).** A 2025 EMNLP Findings / ICML-workshop paper, "Inefficiencies of Meta Agents for Agent Design," challenges the frontier's economics: (1) naïve reuse of prior designs *hurts* meta-learning—dumping all prior agents into context does worse than ignoring them; (2) evolutionary variants raise performance but *collapse behavioral diversity*, undermining complementary ensembling; and (3) design cost is rarely justified—only on **2 of the studied datasets** did total design-plus-deployment cost beat human-designed agents even when amortized over 15,000+ examples [7]. This is the strongest published rebuttal to the "learned beats hand-built" thesis and belongs alongside the AFlow cost numbers.

**Why this matters for orchestrator-vs-blackboard.** These systems reframe the dichotomy as a *search space* choice: fixed-topology orchestrators are one point, blackboards another, and both are reachable/expressible in ADAS's code space or GPTSwarm's graph space. The learned-orchestration literature supplies quantitative evidence on the exact trade-offs the dossier cares about—topology quality over agent count [5], adversarial-agent gating via connectivity [5], code vs. graph expressiveness for conditional control flow [1], per-query adaptivity vs. static design [6], and a sober cost-benefit critique [7]. Omitting it leaves the SOTA claim resting on two hand-built paradigms while an entire "search over coordination structures" axis goes unaddressed.

### Standardized Agent Benchmarks Beyond tau-bench: The Shared Yardsticks for Orchestrator vs. Blackboard/Swarm Systems

### Why the comparison section needs more than tau-bench and one internal eval

Grounding an orchestrator-vs-blackboard comparison on a single LangChain blog plus Anthropic's private research eval is thin: neither is reproducible by third parties, and tau-bench itself is a narrow, saturating target. τ-bench covers only customer-service tool use in two domains (retail, airline), and τ²-bench extends it with 114 telecom tasks and dual-control (user *and* agent can call tools) [1]. Critically, these benchmarks "report a single end-to-end correctness number and do not localize where in the agent pipeline a failure originates" [1] — exactly the wrong instrument for distinguishing a centralized orchestrator from a decentralized blackboard/swarm, where the interesting differences are in coordination overhead, parallelism, and failure propagation. A 2026 dossier should therefore anchor to the broader public benchmark suite below.

### The seven shared yardsticks

**GAIA (General AI Assistants).** 466 real-world questions across three difficulty levels requiring reasoning, multimodality, web browsing, and tool use; 300 are held private to power a Hugging Face leaderboard [2][3]. The framing question — "conceptually simple for humans, hard for assistants" — is captured by the original gap of 92% (human) vs. 15% (GPT-4 with plugins) [3]. By 2025 systems reached ~65% (e.g., H2O.ai h2oGPTe at 65%) [2], and Princeton's HAL leaderboard now hosts GAIA runs [2]. GAIA is the canonical "does the whole assistant work end-to-end" test, and orchestrator/subagent designs are routinely tuned against it.

**AppWorld (ACL 2024).** A high-fidelity execution engine of 9 apps (Amazon, Spotify, Venmo, Gmail, Todoist, SimpleNote, Splitwise, Phone, FileSystem) exposed via 457 APIs over 101 database tables (~370K rows, ~100 simulated people) [4][5]. Tasks span up to 6 apps, with solution code averaging 41–50 lines and up to 26 API calls [4]. Because tasks are inherently multi-app and require iterative, stateful coding, AppWorld is one of the few public benchmarks where task decomposition across sub-agents (an orchestrator dividing work by app) can be measured directly — several 2025 multi-agent papers (e.g., ProST) use it as their target [4].

**WebArena / VisualWebArena.** WebArena's original GPT-4 agent scored 14.41% end-to-end vs. a 78.24% human baseline [6]; by 2026, ColorBrowserAgent reports 71.2%, beating CUGA (61.7%) [6]. VisualWebArena adds 910 multimodal tasks across Classifieds, Shopping, and Reddit, where early top agents managed only 16.4% vs. 88.7% human [6]. WebArena has become "one of the most widely adopted web agent benchmarks," seeding VisualWebArena, WebChoreArena, and the BrowserGym ecosystem [6] — making it a natural common substrate for comparing planner/executor (orchestrator) splits against monolithic agents.

**SWE-bench / SWE-bench Verified / SWE-bench Multimodal.** The de facto software-engineering agent yardstick. On SWE-bench Verified, frontier proprietary systems reach 71–77% [7], and open scaffolds like Live-SWE-agent report 79.2% with Claude Opus 4.5 (Nov 2025) [7]. SWE-bench Multimodal extends the task to visual inputs (screenshots, UI mockups, diagrams) [7]. This family is where the architectural debate is sharpest: agentic scaffolds (single-agent CodeAct-style loops vs. multi-agent orchestration such as localizer/editor/tester roles) are compared head-to-head, and the "Dissecting the SWE-Bench Leaderboards" survey profiles those submitter architectures explicitly [7].

**OSWorld (NeurIPS 2024).** 369 open-ended computer-use tasks in real Ubuntu/Windows/macOS environments with execution-based evaluation [8]. The July 2025 OSWorld-Verified release moved infrastructure to AWS with ~50x parallelization (evaluation in ~1 hour) and cleaned task quality [8]. Scores climbed from single digits to ByteDance Seed 2.1 Pro at 78.8% and Claude Opus 4.6 at 72.7% (Feb 2026, ~84% of the ~72% human baseline) [8]. OSWorld is the standard GUI/computer-use yardstick against which screen-only vs. skill/CLI-mediated (and multi-agent) computer-use designs are compared.

**BrowseComp (OpenAI, April 2025).** 1,266 deliberately hard "find the entangled needle" browsing problems, designed because SimpleQA-style fact retrieval had saturated [9]. The headline result is architectural, not merely tool-access: OpenAI's Deep Research reaches 51.5% vs. 1.9% for GPT-4o-with-browsing — a ~27x gap showing success depends on reasoning/architecture, not tool availability [9]. This is precisely the benchmark Anthropic used to analyze its orchestrator-worker Research system, where "token usage alone explained 80% of performance variance," with tool-call count and model choice the other factors [10]. Anthropic's own claim — a lead-agent + subagent system beat single-agent Opus by 90.2% on their *internal* research eval [10] — is exactly the kind of number that BrowseComp makes externally checkable.

### Tying benchmarks to the architecture debate

Public write-ups frame the orchestrator (supervisor/conductor) vs. blackboard/swarm choice cleanly: in a swarm only one agent is active at a time with decentralized sequential handoffs, while a supervisor routes sub-tasks through a central coordinator with dynamic routing and conflict resolution [11]. The heuristic — orchestrators for auditability/predictability, swarms for scale/fault-tolerance [11] — is untestable on tau-bench alone but becomes measurable on the multi-app (AppWorld), long-horizon web (WebArena, BrowseComp), and repo-scale coding (SWE-bench) benchmarks above.

Finally, the dossier should cite a *meta*-benchmark: the **Holistic Agent Leaderboard (HAL)** from Princeton's SAgE group, "the standardized, cost-aware, third-party leaderboard" that ran 21,730 rollouts across 9 models and 9 benchmarks (coding, web, science, customer service) for ~$40,000 [12]. HAL's three-dimensional analysis over *models × scaffolds × benchmarks* is the correct frame for an architecture comparison — and its counterintuitive finding that more reasoning budget *lowered* accuracy in 21 of 36 settings [12] is a caution against reading any single orchestrator-vs-swarm headline number literally.

### Reasoning Models and the Economics Shift: Why the Dossier's 4x/15x/80% Token Framing Is Stale

### The problem with anchoring cost framing to mid-2025 Anthropic numbers

The dossier's cost model rests on three figures from Anthropic's June 2025 multi-agent engineering post: agents use ~4x the tokens of chat, multi-agent systems ~15x, and token usage alone explains ~80% of performance variance on BrowseComp (95% with tool calls and model choice added) [1]. Anthropic's own conclusion was economic, not architectural: multi-agent "only works for high-value tasks" where task value exceeds token cost, and a single Research session "can use millions of tokens, dollars per query" [1]. Those numbers are real and still directionally useful, but they were measured on **Claude Opus 4 as a near-non-reasoning workhorse**. By 2026 the token-vs-performance calculus that made fan-out attractive has been substantially rewritten by extended-thinking/reasoning models, and the dossier is stale on exactly this axis.

### The core mechanism: reasoning models already spend the tokens that fan-out used to justify

Anthropic's key empirical claim is that multi-agent wins *because* it spends more tokens — token usage explains 80% of variance [1]. But reasoning models are a *different way to spend the same tokens*. A single reasoning agent with a large thinking budget can absorb work that previously required parallel subagents, because test-time compute inside one context is now a first-class dial. o3 introduced adaptive compute where a complex problem gets a much longer chain [2]; DeepSeek-R1-0528 nearly doubled average reasoning from ~12K to ~23K tokens per hard problem [2]. When the performance driver is "spend enough tokens," a single reasoning call and a multi-agent swarm become partial substitutes, not complements.

The most direct 2026 evidence: an arXiv study, *Single-Agent LLMs Outperform Multi-Agent Systems on Multi-Hop Reasoning Under Equal Thinking Token Budgets* [3], argues that reported multi-agent gains are "often confounded by increased test-time computation," and that **when compute is normalized, single-agent systems match or outperform multi-agent systems**. It grounds this in the Data Processing Inequality: under a fixed reasoning-token budget with perfect context utilization, a single agent is more information-efficient, because splitting work across agents and re-serializing through an orchestrator can only lose information, never create it [3]. It also flags measurement artifacts (especially in API-based budget tracking) that inflated prior multi-agent results [3]. This directly undercuts a naive reading of Anthropic's 80%-variance stat: if the variance is really about compute, you can buy it more cheaply inside one reasoning agent than by paying the 15x coordination tax [1][3]. A related "Science of Scaling Agent Systems" line of work [8] and budget-planning papers like FutureWeaver [3] reinforce that compute-normalization is now the required control.

### The economics did not simply get cheaper — they bifurcated

The dossier likely assumes the ~10x/year inference price decline applies uniformly. It does not. Epoch AI finds inference prices fell a median ~50x/year (200x/year post-Jan-2024, ranging 9x–900x by milestone) — but **explicitly excludes reasoning models** from per-token comparisons because they "generate a much larger number of tokens... making these models cost more in total" [7]. Reasoning shifts cost from the (falling) per-token axis to the (rising) tokens-per-task axis. Concretely: thinking tokens bill at the **output** rate [4], the priciest tier. o3's headline $2/$8 per M-token rate hides that reasoning tokens can inflate the actual bill 3–10x [5], and a single thinking call commonly spends 3–10x the tokens of a bare completion [4]. GPT-5-class extended reasoning is quoted around $15 input / $75 output per M tokens [6]. Researchers have named a **"price reversal phenomenon"**: across 8 models and 12 tasks, ~32% of "cheaper" reasoning models end up costing *more* in practice once thinking tokens are counted [6]. So "reasoning made a single agent able to do more" and "reasoning is cheap" are not the same claim — the first holds, the second frequently fails.

### Latency: reasoning changes the UX constraint, not just the bill

The dossier's cost axis should be paired with a latency axis that reasoning models make sharper. Reasoning models generate thinking tokens *before* first answer token, adding 10–150s of first-answer latency [9]; they are "the wrong tool for real-time streaming" and interactive surfaces should route to low-TTFT models [9]. This matters for the orchestrator-vs-blackboard tradeoff: multi-agent fan-out was partly a *latency* play (parallelize to hide per-agent thinking time), and a single deep-reasoning agent reintroduces serial latency even as it cuts coordination overhead. The choice is now a three-way tradeoff among tokens, wall-clock latency, and coordination complexity — not the two-way cost-vs-quality framing in the current dossier.

### The "overthinking tax" caveat cuts against blind reasoning budgets too

Reasoning is not a free substitute for good architecture. Reasoning models "overthink," producing outputs 5–10x longer than instruct models even when both are correct [10], wasting compute on easy problems while helping only on hard ones [10]. OckBench reports same-size 7B reasoning models with similar accuracy differing 3.3x in tokens and 5x in latency — an explicit "Overthinking Tax" [10]. So the updated framing is not "reasoning agents strictly dominate fan-out," but "a well-budgeted single reasoning agent captures much of the compute-driven gain that previously justified multi-agent fan-out, at lower coordination cost — provided the thinking budget is adaptive."

### Field practice has converged, validating the reframe

The 2025 Cognition ("Don't Build Multi-Agents") vs. Anthropic ("Do — carefully") standoff [1] has effectively collapsed by 2026: Cognition itself shipped a coordinator assigning work to isolated managed Devins in March 2026, a middle ground [8]/[3-adjacent coverage]. The practical rule the dossier should adopt: default to a single reasoning agent with an adaptive thinking budget; escalate to multi-agent only when the task is genuinely breadth-first (independent parallel search paths, aggregate context exceeding one window) and the task value clears the now higher-and-lumpier reasoning-token bill [1][3][7].

### Memory Architecture as a First-Class Differentiator: The Persistent-Memory Layer Beneath Blackboard-Style Shared State

### The gap: blackboard "shared state" is only half the story

The dossier's blackboard section treats the shared blackboard as ephemeral working memory — a scratchpad multiple agents read and write during a single run [8]. That framing is faithful to the canonical Hearsay-II blackboard pattern, but it misses what production systems actually do: they bolt a *persistent long-term memory layer* underneath the blackboard so that shared state survives across sessions, users, and time horizons. This persistent layer is not an implementation detail — it is increasingly the primary axis on which architectures compete. Four systems define the current state of the art: **Mem0**, **Letta/MemGPT**, **Zep**, and **A-MEM**. The through-line is that "shared state" and "long-term memory" are the same problem viewed at two timescales, and the blackboard-vs-orchestrator distinction is orthogonal to — and often subordinate to — the choice of memory substrate.

### The blackboard *is* a shared-memory abstraction, and it needs persistence

Recent LLM blackboard work makes the equivalence explicit. In a blackboard multi-agent system (bMAS), the shared blackboard is "the exclusive communication and memory substrate," with a control layer activating agents opportunistically rather than a supervisor micromanaging them [8][9]. Crucially, these papers flag the missing piece themselves: persistent shared state "leaves open how such state should be updated, authorized, and audited during long-horizon collaboration" [8]. The blackboard solves *intra-run* coordination; it says nothing about *inter-run* persistence.

Letta makes the bridge concrete. Its **shared memory blocks** are a database-backed blackboard: multiple agents attach to the same block by `block_id`, and when one agent edits it "all others see the change immediately" [6][7]. This is a literal blackboard whose contents persist beyond the session because they live in a datastore, not just the context window. Letta even surfaces the concurrency problems a naive blackboard hides: `memory_insert` is append-only and concurrent-safe, `memory_replace` is safe only if the target is unchanged, and `memory_rethink` is "last-writer-wins" and unsafe for simultaneous multi-agent writes — so the recommended pattern is a single "owner" agent for heavy edits with others appending [7]. That is exactly the write-arbitration problem the classic blackboard control shell existed to solve, resurfacing in a persistent, multi-agent setting.

### Four architectural strategies for the persistent layer

**Mem0** treats memory as an extraction–consolidation–retrieval pipeline that "fundamentally separates ephemeral conversational context from durable long-term memory" [1][2]. Salient facts are extracted from dialogue, consolidated into a persistent store, and selectively retrieved into context on later sessions. Its graph variant (Mem0[g]) stores memories as directed labeled graphs to capture relations, adding ~2% over the base config [1][2]. Reported efficiency is the headline claim: ~91% lower p95 latency and >90% token savings versus full-context [1][2].

**Zep** (built on the open-source **Graphiti** engine) models memory as a *temporal knowledge graph* with a three-tier hierarchy — an episodic subgraph (raw, non-lossy messages), a semantic-entity subgraph (extracted entities and relationship edges), and a community subgraph (clustered summaries) [3][4]. Its distinguishing feature is a **bi-temporal model**: it tracks both when an event occurred (T) and when it entered the system (T′), and it *invalidates* facts by setting expiration timestamps when new information contradicts old edges — preserving history while prioritizing current truth [3][4]. This directly addresses the blackboard's blind spot: a plain shared-state store overwrites, whereas Zep reconciles contradictions and maintains an audit trail, which is why it is positioned for regulated industries [3][14].

**Letta/MemGPT** implements an OS-style memory hierarchy: in-context "core memory" blocks (RAM) versus out-of-context archival/recall memory (disk), with the agent paging data between tiers via tool calls [5][6]. Its **sleep-time agents** run in the background during idle periods to reflect on history and rewrite shared memory blocks asynchronously — notably, the primary agent often *lacks* memory-edit tools, which are delegated to the sleep-time agent so consolidation happens off the critical path [10]. This is a persistent-memory analog to a blackboard "knowledge source" that curates the board between problem-solving cycles.

**A-MEM** attacks the *schema* problem. Prior systems rely on "predefined memory access patterns specified in the workflow"; A-MEM instead builds Zettelkasten-style atomic notes (content + LLM-generated keywords, tags, contextual description), auto-links semantically related notes, and — critically — *evolves* existing notes when new memories arrive, updating their contexts and tags rather than leaving them static [11]. This makes the persistent store self-organizing, avoiding the brittleness of a fixed blackboard schema.

### The real production differentiator: governance, not just recall

The most important point for an orchestrator-vs-blackboard framing is that persistent shared memory introduces problems the blackboard model never had to answer: **authorization, provenance, and audit** [8][12]. Once shared state outlives a run and spans users, an uncontrolled blackboard becomes a data-leakage attack surface. The **Collaborative Memory** framework formalizes the fix: private vs. shared memory tiers, immutable per-fragment provenance (contributing agents, accessed resources, timestamps), and time-evolving bipartite access-control graphs (user→agent, agent→resource) that filter reads and route writes with redaction/anonymization [12]. Surveys of multi-agent memory reach the same conclusion — the open problem is "bridging ephemeral structural adaptations with persistent collective knowledge," with access control, provenance, and governance as first-class concerns [13]. Systems like MIRIX (six memory types, each managed by a dedicated agent under a meta-controller) and G-Memory (three-tier graph of strategic/task/communication knowledge) show the design space expanding well beyond a flat board [13].

### Caveat: benchmarks are contested

Vendor-reported numbers diverge sharply from independent replications. Mem0 cites ~91.6% on LOCOMO, but independent evals put the reproducible figure closer to 58–66%; Letta lands around ~83% (the highest verified classic open-source score); and Zep published a rebuttal correcting its own LOCOMO score to 75.14% [14]. The practical guidance across comparisons is consistent: benchmarks (LOCOMO, LongMemEval, BEAM) disagree, so evaluate on your own workload, and pick by use case — Mem0 for consumer apps, Letta for autonomous agents, Zep for regulated/temporal-reasoning domains [14].

### Bottom line for the dossier

The blackboard section should be extended to state that production blackboard-style systems are defined less by their *control regime* (opportunistic vs. supervised) than by their *persistence substrate*. Shared state and long-term memory are one continuum; the differentiators are graph vs. pipeline vs. OS-hierarchy vs. self-organizing storage, temporal reconciliation (fact invalidation), asynchronous consolidation (sleep-time compute), and — decisively for enterprise — access control, provenance, and audit over that persistent shared memory.

The reviewer is correct that the number is not derivable from its own stated premise. The reviewer's counter-derivation is the physically correct one: if the orchestrator's decomposition/assignment decision is a **serialized 3-second LLM call**, the hub can emit **one decision per 3 seconds = ~0.33 tasks/sec**, regardless of how many workers idle behind it. Adding workers does not raise hub throughput when the hub is the serial bottleneck — that is the entire point of calling it a bottleneck. To actually sustain 6.7 tasks/sec through a single serialized hub, each hub call would need to be **~0.15 s (1 ÷ 6.7)**, 20× faster than the stated 3 s. The two premises (3 s call, 6.7 tasks/sec) are mutually exclusive under serial execution.

(The only way to rescue "6.7/sec" is to abandon the "serialized hub" framing entirely — e.g. assume the 3 s call assigns a *batch* of 20 tasks at once, giving 20 ÷ 3 ≈ 6.7 tasks/sec of *assignment* throughput. But that reinterpretation contradicts the surrounding text, which invokes the number precisely to illustrate the orchestrator as a *serial* "single point of failure" and "throughput bottleneck." Under that intended reading, the correct ceiling is ~0.33 tasks/sec.)

- **36.94%** is the category's share in the paper's primary overall distribution of annotated failures (Specification 41.77% / Inter-Agent Misalignment 36.94% / Verification 21.30%) [6].
- **32.3%** appears in a different slice/subset analysis of the same taxonomy [3].

Neither is a "single failure mode" figure; both are category-level. The single largest *mode* within FC2 is Reasoning-Action Mismatch at 13.2% [1] — that would be the correct number to compare against the 11–16% single-mode figures if a like-for-like comparison is wanted.

#### Bottom line
- "Step repetition 15.7%," "termination-unawareness 12.4%," and **"spec-disobedience 11.8%" are all verified** against MAST Figure 1 [1] — the reviewer can drop the doubt about 11.8%.
- The reviewer's substantive point stands: **"inter-agent-misalignment failures dominate" and its ~32–37% figure describe a whole CATEGORY (FC2), not a single mode**, so presenting it inline with single-mode percentages is misleading and should be relabeled (e.g., "the Inter-Agent Misalignment *category* accounts for ~37% of failures," or compare like-for-like using FM-2.6 Reasoning-Action Mismatch at 13.2%).

## 13. Conclusion

Orchestrator and blackboard are not rivals so much as two answers to one question — *where control lives and how much agents must see each other's work* — and the right answer is task-shaped. Orchestrators win when the work decomposes cleanly, when breadth-first parallelism pays, and when a single traceable control point is worth its message-restatement tax (auditability, HITL gating, durable interrupt/resume, predictable cost). Blackboards win when structure emerges at runtime, sub-results are interdependent, and mid-flight reactivity beats serialized re-planning. The published evidence refuses a universal winner: Anthropic's orchestrator crushed single-agent research by 90.2% yet fails on interdependent coding; a swarm beat a supervisor on τ-bench-with-distractors; blackboard systems post 13–57% end-to-end gains at ~3.5× fewer tokens on the tasks that suit them.

Three 2026 realities should temper any strong claim. First, **reasoning models partially substitute for fan-out** — when token spend explains ~80% of multi-agent's edge, a well-budgeted single reasoning agent captures much of that gain at lower coordination cost, so multi-agent should be an escalation, not a default. Second, **the protocol substrate biases the topology**: MCP wires hubs, A2A wires peers, and a true blackboard needs a shared-state/broker layer bolted on. Third, the **bounded/controlled blackboard** — a reactive shared state under a deterministic control unit and gate — is a genuine, under-explored middle path: it grafts the orchestrator's reproducibility, cost bounds, and WORM auditability onto the blackboard's reactivity, at some cost to emergent flexibility. It is a design position to evaluate, not a proven winner.

The durable guidance: default to a single reasoning agent for dependent work; use a context-owning orchestrator for decomposable/breadth-first work; reserve shared-state collaboration for genuinely interdependent, cross-checking tasks; and whatever you choose, impose hard iteration caps, keep an LLM out of the final gate, instrument tokens/latency/traces uniformly, and benchmark under compute normalization on public yardsticks rather than a single vendor eval.
