# Improvement plan — UX journey & technical roadmap

_Produced 2026-07-02 by a 4-lens adversarial review (UX journey simulation, visual design, tech architecture, product positioning; 37 code-verified findings, 31 proposals) + synthesis. Owner's complaint driving it: **the UI journey is still confusing for non-technical users.**_

## The diagnosis

The dashboard inverts its own funnel: it front-loads ~330 words of pedagogy and 15 expert controls before anything moves, then delivers its entire payoff (same PC build, 12 vs 7 vs 5 calls) as an unstated 9-column table below the fold — the user does all the work and the product never says its own punchline. The default mode literally announces itself as "fake" while a committed recording of real Claude calls (cassettes/demo.json) sits unused, so the most trustworthy version of the demo is hidden behind jargon ("Cassette"). The ELI5/Expert toggle is cosmetic — it swaps two intro paragraphs and leaves the meters, feeds, controls, and table in expert vocabulary, so choosing "simple" never actually simplifies anything. During a run, three concurrent panels with ~15 live-updating regions give the eye no focal point, and at the end the only event is the SSE connection silently closing. All of this is compounded by a structural tax: the entire front end is a 400-line Python raw string (live.py:285-686) that no linter or test can touch, which is why every journey iteration so far has been slow, partial, and drift-prone.

## North-star journey (the target experience)

**The ideal first visit (non-technical, zero clicks to value):**

1. **0s — Load.** Page opens in **Simple mode** (default). One headline: "One team decision, three ways to run the meeting — watch which one wastes the least." Below it, one line: the 🎯 target ("tier 3 · $900 · 550W · high FPS"). No control bar wall — just a collapsed "⚙ Customize this run" and a collapsed "👋 First time here? The 60-second tour".
2. **~1s — It starts by itself.** With no URL params, the page auto-selects the recorded real run (cassette) and auto-runs after an 800ms beat. Status line: "▶ Watching a recorded real run — 4 AI teammates spec this PC with real Claude replies. Free to replay."
3. **1–20s — The race is legible.** The hero visual is ONE shared race-lane chart: three horizontal lanes (one per engine, engine-colored) over a common "agent turns" axis. Each finished call drops a dot — filled = changed something, hollow = wasted turn. The orchestrator's hub tax becomes a visible trail of hollow dots. A 🏁 lands where each gate passes. Live "turns · $" counters tick in a sticky score strip so the payoff is on screen at every scroll position. (Detailed panels exist behind "Show the machinery ▾".)
4. **~20s — The verdict is stated, not inferred.** On all_done, a banner appears ABOVE everything and scrolls into view: "🏆 Same PC — three price tags. All three landed on the identical build (tier 3 · $900 · 550W · high FPS). Hybrid: 5 turns · Blackboard: 7 · Orchestrator: 12 — 2.4× more talking for the exact same answer." Winner highlighted in its engine color, numbers 28px.
5. **20–90s — Explore, one door at a time.** Under the banner: a 4-column simple table (Way | Turns taken | Wasted turns | Cost), winner row tinted, with "Why did the orchestrator talk more? →" linking to the explanation. A ↺ Replay button re-runs instantly. "⚙ Customize" opens GPU tier / budget and a big "▶ Run the race" button.
6. **Expert path.** Flipping the global **Simple | Expert** toggle (persisted) reveals exactly today's UI: mode/model pickers, engine checkboxes, flow SVGs, boards, feeds, the 9-column table, Glossary tab. Nothing is dumbed down — it is deferred.

Every step exists as plumbing today (auto-run param, cassette, event handlers, seg component, dead .howto CSS); this journey is re-sequencing, not new machinery.

## Success criteria (measurable)

1. **The 90-second grandma test.** A non-technical person given only the URL can, within 90 seconds and with **zero clicks**, answer: "Did the three ways get the same answer?" and "Which way wasted the most?" (Pass = they quote the verdict banner, not the table.)
2. **Time-to-verdict ≤ 20s from page load** with no interaction (auto-run + cassette at default pace), measured stopwatch from load to verdict banner visible.
3. **Decisions before value = 0** (currently ~10: mode, model, 3 checkboxes, 2 inputs, slider, tab, Run). Controls visible in Simple mode before first run ≤ 3 (Run, Simple|Expert, Customize disclosure).
4. **The verdict is stated in a sentence** — grep the served page after all_done for the banner text; the Playwright smoke asserts banner presence + "12", "7", "5" + identical plan cells.
5. **Forbidden-word check (CI-greppable):** "fake", "prompt", "Cassette" (user-facing), "$0.00000" appear nowhere in served UI copy.
6. **Expert parity:** every datum visible today (9-column table, flow SVGs, boards, feeds, glossary, mode/model controls) remains reachable in ≤ 2 clicks from load (Expert toggle, tab). No information deleted — only deferred.
7. **Slider sanity:** dragging the pace control right makes the run finish sooner.
8. **Engineering:** `ruff check` and pytest pass in CI on every push (no `|| true`); the front end lives in .html/.css/.js files a formatter can touch; changing task.py's GPU_PRICE breaks zero UI numbers (server-computed expected_plan).
9. **Shareability (post-P2):** README opens with a moving GIF of the race; a GitHub Pages link replays the recorded run with no install; repo is public under a name whose first Google page it owns.

### Lens 3: Technical architecture & engineering quality

_Engineering fundamentals are genuinely good — a clean 2,400-line src tree, a real Harness abstraction where the three engines override only run(), fairness enforced in code, and 10 deterministic offline tests.
